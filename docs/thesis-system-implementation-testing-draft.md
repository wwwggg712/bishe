# 5 系统实现

## 5.1 开发环境与技术选型

本系统采用前后端分离的实现方式，前端使用 Vue 3、Vite、Pinia、Vue Router 和 ECharts 进行页面构建与数据可视化，后端使用 Flask、Flask-JWT-Extended、Flask-SQLAlchemy 和 APScheduler 构建认证、业务接口与任务调度模块。系统在 Windows 环境下完成开发与验证，后端默认使用 SQLite 作为本地演示数据库，以保证毕业设计答辩场景下的快速部署与稳定运行。

从实现角度看，选择 Vue 3 作为前端框架，主要原因是其组件化能力较强，便于按商家端、用户端和管理员端进行模块拆分；选择 Flask 作为后端框架，则是因为其结构清晰、开发成本低、适合在有限周期内快速构建 RESTful API。与此同时，系统将行为日志分析逻辑封装到独立服务层中，使页面展示逻辑与分析计算逻辑分离，保证了系统的可维护性和可扩展性。

## 5.2 系统总体实现结构

系统总体实现分为表现层、接口层、业务服务层和数据层四个部分。

表现层主要由前端页面组成，负责登录认证、角色导航、图表展示和用户交互。接口层由 Flask 提供统一的 RESTful API，对外暴露认证、日志模拟、统计分析、趋势预测、推荐服务和任务管理等接口。业务服务层封装认证服务、日志模拟服务、统计分析服务、预测服务、推荐服务、策略建议服务以及大模型摘要服务。数据层负责保存用户、商品、任务和聚合指标等结构化数据，同时在当前最小实现中通过内存日志存储与本地数据库结合的方式完成演示闭环。

后端在应用初始化时统一注册蓝图、数据库连接、JWT 认证和定时任务，并自动写入演示数据。该实现方式保证了系统首次启动后即可完成账号初始化、商品初始化和任务初始化，降低了部署复杂度，提高了演示效率。

## 5.3 后端模块实现

### 5.3.1 应用工厂与接口组织

后端采用 Flask 应用工厂模式组织代码。在应用启动过程中，系统首先读取配置项，然后初始化数据库、跨域组件和 JWT 鉴权模块，随后注册认证、分析、预测、推荐、日志模拟、管理员和用户画像等蓝图，最后完成数据表创建、演示数据写入与任务注册。该模式使系统具备较好的模块解耦能力，也为后续扩展不同环境配置提供了基础。

在接口组织上，系统采用按领域划分的方式构建接口模块，主要包括 `auth`、`analytics`、`prediction`、`recommendation`、`simulation`、`strategy`、`llm`、`users` 和 `admin` 等蓝图。每类接口只负责接收请求参数、完成权限判断并调用对应服务层方法，具体计算逻辑统一放在服务层中实现，避免路由函数承担过多业务逻辑。

### 5.3.2 用户认证与角色权限实现

系统采用基于 JWT 的身份认证方式。用户在登录页面输入用户名和密码后，后端调用认证服务校验账号信息，校验成功后生成访问令牌并返回当前用户的角色信息。前端在获取令牌后将其保存到 Pinia 状态中，并在后续请求中自动携带到请求头内。

在权限控制方面，系统根据用户角色划分为管理员、商家和普通用户三类角色。路由层通过元信息判断目标页面允许的访问角色，后端接口则进一步通过 JWT 中的角色字段进行二次校验。采用这种前后端双重校验方式，可以避免仅依赖前端拦截导致的越权访问问题。

### 5.3.3 日志模拟模块实现

为了避免系统退化为“导入静态 CSV 后展示图表”的简单报表系统，本文实现了独立的日志模拟模块。该模块通过预设用户、商品、地区、行为类型和时间信息生成电商用户行为日志，行为类型包括浏览、点击、收藏、加购和购买五类。系统支持一次性生成日志，也支持模拟任务的启动、停止和任务状态查看，从而满足答辩时的动态演示需求。

从实现细节看，日志模拟服务会优先确保基础数据已经完成初始化，然后按设定批次生成行为日志并写入内存日志存储结构。管理员既可以通过后台任务直接触发一轮模拟，也可以在任务管理页中启动或停止持续模拟任务。这样一来，系统中的统计分析、推荐和异常检测就不再依赖外部静态文件，而是由内部日志流驱动完成。

### 5.3.4 统计分析模块实现

统计分析模块是系统的核心实现部分之一。该模块主要负责对行为日志进行聚合处理，输出总览指标、转化漏斗、地区分布、类目热点、热销商品、冷门商品以及用户 RFM 分层结果。系统在分析服务中首先对日志行为进行计数，然后按照商品、地区和类目进行分组统计，最终生成可直接用于前端展示的结构化结果。

在总览指标实现中，系统统计 PV、UV 和购买次数等核心指标，并构建浏览、点击、收藏、加购、购买五个步骤的漏斗数据。对于商品分析，系统依据行为日志中的商品编号和商品名称统计行为次数，并根据行为权重计算商品热度分。对于用户价值分析，系统根据购买频次、消费金额和最近行为时间实现 RFM 分层，将用户划分为高价值用户、潜力转化用户和待激活用户，从而为商家端经营决策提供支持。

### 5.3.5 趋势预测与异常检测实现

趋势预测模块基于时间窗口对商品热度进行对比分析。系统将近期窗口与基线窗口中的行为得分进行比较，根据二者差值判断商品趋势标签为上升、下降或平稳。这里使用的得分规则与热度模型一致，即为浏览、点击、收藏、加购和购买分别赋予不同权重。该方法实现简单、可解释性强，适合作为毕业设计中的轻量预测方法。

异常检测模块同样基于窗口比较思想实现。系统重点识别两类异常：一类是商品近期流量明显上升但转化率没有同步提升，另一类是特定地区近期浏览量出现明显波动。对于前者，系统会输出高流量低转化预警；对于后者，系统会输出地区流量异常预警。为了便于前端展示，异常结果中还包含异常类型、目标对象、变化幅度和预警等级等字段。

### 5.3.6 推荐、画像与策略建议实现

推荐模块主要面向普通用户，依据用户行为日志提取其偏好类目、行为偏好、价格偏好和活跃时间段，然后结合商品热度信息生成推荐结果，并输出推荐理由。与单纯展示热门商品不同，系统在返回推荐结果时明确给出“为何推荐”的解释，从而提升了系统的可解释性。

用户画像模块面向当前登录用户构建偏好画像，输出内容包括高频类目、主要行为类型、最近行为记录、参与度得分、活跃时段和价格带标签等。通过该模块，用户端页面可以直接展示“你偏好什么”“你通常何时活跃”“你属于哪类消费风格”等结果。

策略建议模块主要面向商家端。系统结合商品浏览量、购买率和异常检测结果，输出高浏览低转化商品、热销机会商品以及建议动作，例如优化详情页、增加活动曝光、优先补货和调整促销策略等。该模块使系统不再停留于“分析是什么”，而是进一步回答“下一步应该怎么做”的问题。

### 5.3.7 大模型报告增强实现

为了增强系统展示效果，本文在后端中加入了大模型报告接口，但该模块并不直接替代统计分析和推荐算法，而是基于结构化分析结果生成自然语言摘要。当前实现中，大模型模块采用安全降级策略：即使外部模型服务不可用，系统也可以根据传入的商品热度、趋势和转化率自动生成说明性文字，保证接口稳定返回结果。

这种设计能够体现系统的创新性，同时避免答辩时因外部接口不稳定而影响整体演示。换言之，大模型在本系统中只是解释层和增强层，真正支撑分析结果的仍然是底层统计与规则模型。

## 5.4 前端模块实现

### 5.4.1 前端路由与状态管理实现

前端基于 Vue Router 构建三角色路由体系，所有业务页面均挂载在统一布局组件下。系统配置了登录页、商家首页、用户首页、推荐页、画像页、管理员总览页、任务管理页和用户管理页等路由，并通过路由元信息标记可访问角色。未登录用户访问业务页面时会被重定向到登录页，已登录用户访问不属于自身角色的页面时会被重定向到首页，从而保证页面访问的安全性。

状态管理方面，系统使用 Pinia 维护当前登录令牌、用户信息和角色信息。这样不仅降低了组件间传值复杂度，还便于 Axios 请求拦截器统一读取令牌并自动附加到请求头中。

### 5.4.2 商家端页面实现

商家端页面围绕“经营分析与决策支持”展开实现。商家登录后可进入经营总览页，页面中集中展示今日经营简报、核心指标卡片、热销机会、滞销风险、异常预警、热销商品、冷门商品、类目热点、用户价值分层、转化漏斗和策略建议等内容。

其中，热销商品和冷门商品通过行为热度排序结果展示当前值得重点关注或需要重点处理的商品；类目热点用于反映当前消费行为较活跃的商品类别；用户价值分层用于展示高价值用户与潜力用户的分布情况；策略建议区则直接给出商品运营动作。通过这一页面，商家不仅可以看到“哪些商品最热”，还可以理解“为什么热”“哪里有风险”“应采取什么措施”。

### 5.4.3 用户端页面实现

用户端页面主要包括推荐首页、推荐中心和个人画像页。推荐首页展示系统根据用户偏好生成的商品推荐结果以及趋势上升商品信息，推荐中心则进一步突出推荐理由和热门趋势，个人画像页展示用户偏好类目、行为偏好、活跃时段、近期行为和价格偏好等结果。

与传统电商推荐页面相比，本文实现更强调可解释性。页面不是简单给出推荐列表，而是把画像标签与推荐原因显式展示出来，使用户能够理解系统推荐依据，从而增强功能完整性和论文中的方法支撑性。

### 5.4.4 管理员端页面实现

管理员端页面包括系统总览页、任务管理页和用户管理页。系统总览页用于展示平台指标、任务状态、系统日志和异常预警信息，帮助管理员快速了解当前运行情况。任务管理页除了支持手动执行后台任务外，还支持模拟任务的启动、停止以及模拟任务状态查看，用于控制演示数据的动态变化。用户管理页则展示当前账号列表，便于管理员验证多角色数据是否正常。

管理员端的存在提升了系统的完整性，使本项目不再只是单一的数据分析页面，而是具备基础运营控制和任务调度能力的完整系统。

## 5.5 关键算法实现说明

### 5.5.1 商品热度评分模型

系统使用加权评分法计算商品热度分，其计算公式为：

`HotScore = 1 × view + 2 × click + 3 × favorite + 5 × cart + 8 × purchase`

之所以采用该模型，是因为不同用户行为对购买意愿的体现程度不同。其中浏览行为仅代表初步兴趣，因此权重最低；购买行为直接代表成交，因此权重最高；收藏和加购则处于中间层级。该模型具有计算简单、含义明确和可解释性强的特点，非常适合在毕业设计中作为核心分析指标。

### 5.5.2 漏斗分析方法

系统以浏览、点击、收藏、加购和购买构建行为转化漏斗。通过对各阶段行为数量进行统计，可以反映用户从接触商品到最终成交的路径变化情况。当某一阶段数量明显下降时，说明对应环节存在潜在转化问题。例如浏览量较高但购买量较低时，往往意味着商品详情页、价格策略或活动设计需要优化。

### 5.5.3 RFM 用户价值分层方法

RFM 分析从最近一次活跃时间、购买频次和消费金额三个维度刻画用户价值。本文在最小实现中重点使用购买频次与消费金额两个指标作为划分依据，并将用户划分为高价值用户、潜力转化用户和待激活用户。该方法能够较直观地反映用户对平台的贡献程度，为商家制定差异化运营策略提供支持。

### 5.5.4 时间窗口趋势判断方法

趋势判断采用窗口对比法，将最近时间窗口中的行为得分与前一窗口得分进行比较。若当前窗口得分高于基线窗口，则认为商品热度呈上升趋势；若低于基线窗口，则认为热度下降；若二者相等，则视为平稳。该方法虽然没有采用复杂的时间序列模型，但具有实现稳定、运算开销小和结果可解释等优点，适合毕业设计演示场景。

## 5.6 本章小结

本章从开发环境、系统结构、后端模块、前端模块和关键算法五个方面介绍了系统的具体实现过程。通过前后端分离架构、日志模拟机制、统计分析模型、推荐与画像模块以及任务控制模块的协同配合，系统完成了从行为日志生成到分析结果展示，再到策略建议输出的完整实现闭环。与原有“导入 CSV 后展示仪表盘”的方案相比，当前系统在工程完整性、业务闭环和分析深度方面均有明显提升。

# 6 系统测试与结果分析

## 6.1 测试目标与测试环境

系统测试的目标主要包括三个方面：一是验证三角色登录、日志模拟、分析计算、推荐与策略建议等核心功能是否能够正常运行；二是验证前后端之间的数据交互是否正确；三是验证系统在本地答辩环境下是否具备稳定构建和稳定演示能力。

测试环境为 Windows 平台，后端基于 Python 3.9 与 Flask 运行，前端基于 Node.js 与 Vite 运行。本地数据库采用 SQLite。测试工具方面，后端使用 Pytest 对服务接口和业务逻辑进行自动化测试，前端使用 Vitest 对路由与关键页面进行组件测试，并通过 Vite 构建命令验证系统能否正常产出前端静态资源。

## 6.2 测试方案设计

根据系统实现特点，本文采用“后端接口测试 + 前端页面测试 + 构建验证”的综合测试方案。

后端接口测试主要覆盖以下内容：登录认证接口、日志模拟接口、总览分析接口、细分分析接口、趋势预测接口、推荐接口、策略建议接口、用户画像接口、异常检测接口、管理员任务接口以及大模型报告接口。前端页面测试主要验证登录、路由守卫、商家端页面、用户端页面和管理员端页面是否能够正确渲染关键内容。构建验证则通过前端生产打包命令检查系统是否具备部署条件。

这种测试方案既覆盖了系统的核心功能，又能够体现毕业设计项目中“从模块正确到系统可交付”的完整验证思路。

## 6.3 后端功能测试

### 6.3.1 登录认证测试

登录认证测试主要验证用户输入正确账号和密码后，后端是否能够返回 JWT 令牌及用户角色信息；当请求未携带令牌时，受保护接口是否能够正确返回未授权状态。测试结果表明，系统能够正确完成登录认证流程，并能够对未授权访问进行拦截，说明认证模块实现正确。

### 6.3.2 日志模拟与任务测试

日志模拟测试主要验证系统是否能够生成行为日志、是否能够返回生成数量和日志预览结果，以及管理员能否查看模拟任务状态并控制模拟任务启动和停止。测试结果表明，系统能够正常生成演示日志，并支持任务状态查询与任务控制，说明日志模拟链路可用。

### 6.3.3 统计分析测试

统计分析测试重点验证总览指标、转化漏斗、地区分布、类目热点、热销商品、冷门商品和用户 RFM 分层等接口返回结构是否正确。测试结果表明，各分析接口均能够返回预期字段，说明统计分析模块具备稳定输出能力。

### 6.3.4 智能分析测试

智能分析测试主要包括趋势预测、异常检测、用户画像、推荐理由和商家策略建议等内容。测试结果表明，趋势接口能够返回上升、下降和平稳标签，异常接口能够根据等级过滤预警结果，用户画像接口能够输出偏好类目、活跃时段和价格带信息，推荐接口能够返回推荐理由，策略接口能够返回针对商品的运营建议。

### 6.3.5 大模型摘要测试

大模型摘要测试重点验证在不依赖外部模型服务的情况下，系统是否仍能返回包含商品名称、热度分、趋势和转化率的摘要文本。测试结果表明，该接口能够稳定返回兜底摘要内容，说明增强层设计具备较好的可用性。

## 6.4 前端页面测试

### 6.4.1 路由与权限测试

前端路由测试主要验证登录态保存、角色识别、页面重定向和访问拦截等逻辑。测试结果表明，未登录用户访问业务页面时会被正确跳转到登录页，已登录用户访问非本角色页面时也会被拦截，说明路由守卫逻辑有效。

### 6.4.2 商家端页面测试

商家端页面测试主要验证经营总览、今日经营简报、热销机会、滞销风险、异常预警、热销商品、冷门商品、类目热点、用户价值分层、转化漏斗和策略建议等内容是否能够正确渲染。测试结果表明，商家端能够消费真实分析接口并展示关键经营信息，说明商家页面功能已形成闭环。

### 6.4.3 用户端页面测试

用户端页面测试主要验证推荐首页、推荐中心和个人画像页是否能够显示推荐商品、趋势商品、推荐理由、画像标签和近期行为信息。测试结果表明，用户端能够正常展示推荐与画像结果，说明用户功能链路运行正常。

### 6.4.4 管理员端页面测试

管理员端页面测试主要验证系统总览、任务状态、系统日志、用户列表和模拟任务控制是否能够正确展示。测试结果表明，管理员可以在任务管理页中看到模拟任务控制区、启动与停止按钮以及任务列表信息，说明管理员端已具备运行控制能力。

## 6.6 测试结果分析

通过对测试结果的分析可以看出，系统的核心功能已经能够稳定运行。后端接口测试覆盖了认证、日志模拟、统计分析、趋势预测、推荐、画像、策略建议、异常检测、任务控制和大模型摘要等关键模块，说明系统底层业务逻辑较为完整。前端测试覆盖了商家端、用户端和管理员端三类页面以及路由逻辑，说明系统前端展示链路与权限链路均能正常工作。

与此同时，前端构建虽然成功，但仍存在两个非阻塞告警：一是第三方依赖中的注释在 Rollup 构建阶段被移除，二是打包后主资源体积偏大。这些问题不会影响当前答辩演示，但也说明系统后续仍可在按需加载、组件拆分和构建优化方面继续完善。

## 6.7 系统不足与改进方向

尽管当前系统已经完成最小可交付版本，但仍存在一定不足。

第一，当前日志主要来源于系统模拟生成，尚未接入真实电商平台日志数据，因此在数据真实性和规模上仍有提升空间。第二，趋势预测方法以轻量窗口比较为主，尚未引入更复杂的时间序列模型。第三，推荐模块目前以规则与画像驱动为主，尚未进一步引入协同过滤或深度学习推荐方法。第四，前端构建包体积仍偏大，说明系统在工程优化层面还有继续提升的空间。

后续可从以下几个方面进行改进：接入真实采集或半真实日志数据源；完善商品管理和运营动作闭环；引入更丰富的预测模型和推荐算法；对前端进行路由级代码分割和图表组件按需加载；扩展大模型模块，使其支持日报、周报和多维经营分析总结。
