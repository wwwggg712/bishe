# 基于用户行为日志的电商智能分析与可视化系统设计与实现

摘要：电商平台在日常运行过程中会持续产生浏览、点击、收藏、加购、购买等大量行为数据。
传统业务系统虽然能够记录这些数据，但在课程设计和中小型应用场景中，往往存在数据链路分散、分析结果碎片化、角色视角割裂、展示方式单一等问题。
围绕这一问题，本文设计并实现了一套基于用户行为日志的电商智能分析与可视化系统。
系统采用前后端分离架构，后端基于 Flask、Flask-SQLAlchemy 与 Flask-JWT-Extended 构建 RESTful 接口服务，前端基于 Vue 3、Vite、Pinia 和 ECharts 实现多角色交互界面与可视化看板。
系统以 MySQL 作为统一的数据存储，用户、商品、任务等结构化业务数据与行为日志均持久化保存；其中行为日志落库到行为日志表 `behavior_logs`，支持分页预览与手动清理（保留最新 N 条）以控制数据增长。
系统面向管理员、商家和普通用户三类角色，构建了“日志生成与记录—分析计算—结果解释—可视化展示”的闭环：管理员侧在任务管理中完成批量模拟生成与清理，并在日志预览中分页查看日志；商家侧提供经营总览、事件级转化指标、地区与类目热点、热销与冷门商品、用户活跃信号、简化用户分层和异常预警，并可基于可配置大模型（如 DeepSeek）生成经营建议；用户侧提供基于真实行为的推荐、用户画像与近期行为反馈，并支持大模型生成解释性摘要。
本文在算法实现上并未刻意追求复杂模型堆砌，而是结合毕业设计场景，采用行为权重评分、偏好类目提取、热度排序、时间窗口对比和简化分层等可解释方法，并在推荐理由中显式给出热度分与偏好得分等证据化信息，保证系统结果可复核、可展示、可答辩。
测试结果表明，系统能够稳定完成三角色业务闭环，后端核心测试与前端界面测试均通过，系统具备较好的完整性、可用性与演示可信度。

关键词：用户行为日志；电商分析；可视化系统；用户画像；推荐系统；Flask；Vue 3；ECharts

Abstract: E-commerce platforms continuously generate large volumes of behavioral data such as browsing, clicking, favoriting, adding to cart, and purchasing.
Conventional business systems can record such data, yet in graduation project scenarios and lightweight applications, data processing links are often fragmented, analytical results are isolated, role perspectives are disconnected, and visual presentation remains insufficient.
To address these issues, this paper designs and implements an e-commerce intelligent analysis and visualization system based on user behavior logs.
The system adopts a front-end/back-end separated architecture: the back end is implemented with Flask, Flask-SQLAlchemy and Flask-JWT-Extended to provide RESTful APIs, while the front end is built with Vue 3, Vite, Pinia and ECharts to support multi-role interaction and dashboards.
MySQL is used as the unified data store; structured business entities and behavior logs are persisted in the database.
Behavior logs are stored in the `behavior_logs` table, supporting paginated preview and manual retention (keeping the latest N logs) to control data growth.
The system serves three roles—administrator, merchant and customer—and establishes a closed loop from log generation and recording to analytical computation, result interpretation and visual presentation.
The administrator manages batch simulation and retention in the task console, and previews logs with pagination.
The merchant side provides business overview, event-level conversion indicators, regional and category hotspots, hot and cold products, recent user activity signals, simplified segmentation and anomaly alerts, and can generate operational suggestions with a configurable LLM (e.g., DeepSeek).
The customer side provides recommendations, profiling and recent-action feedback based on real user behaviors, and supports LLM-generated explanatory summaries.
Instead of pursuing overly complex models, this study adopts interpretable methods including behavior-weighted scoring, preference category extraction, hotness ranking, time-window comparison and simplified segmentation; recommendation reasons also expose traceable scores to improve defensibility.
Test results show that the system stably completes the business loop for all three roles, and both back-end and front-end tests pass successfully.

Keywords: user behavior logs; e-commerce analytics; visualization system; user profiling; recommendation system; Flask; Vue 3; ECharts

## 1 绪论

### 1.1 研究背景

电子商务平台已经成为网络消费的重要载体。用户在平台中的每一次浏览、点击、收藏、加购和购买行为，都会形成可追踪的行为日志。这些日志不仅反映了用户对商品、类目和价格区间的偏好，也直接关联着商家运营决策、平台推荐效果和营销资源投放效率。对于成熟的商业平台而言，行为日志通常会被用于推荐排序、用户分层、热度监测、活动评估和异常识别。对于教学演示、课程设计和中小型系统开发而言，问题不在于“有没有数据”，而在于“能否把行为日志组织成一条看得见、讲得清、跑得通的分析链路”。

从实际应用看，很多电商类课程项目更重视交易流程或商品展示，用户行为日志往往只停留在数据库记录层面，没有进一步形成统一的数据处理闭环。部分系统只展示静态图表，缺少行为依据与解释逻辑；部分系统虽然引用了推荐、画像、用户分层等概念，却没有对结果来源作出说明，导致展示效果与实际实现之间存在落差。毕业设计答辩对系统的要求不仅是“页面能运行”，还强调“业务逻辑讲得通、数据来源说得清、分析结果有依据”。如果系统一味堆叠术语而缺乏可解释性，老师往往会从算法来源、指标口径、角色边界和数据可信度几个方向提出质疑。

本课题正是在这样的背景下提出。本文围绕电商用户行为日志，设计一套面向毕业设计场景的智能分析与可视化系统，不追求大而全的工业级复杂架构，而是以“真实业务闭环、合理技术选型、分析结果可解释”为核心。系统既要覆盖管理员、商家和普通用户三种角色，也要把日志生成、行为记录、画像构建、推荐输出、经营分析和结果展示有机连接起来，使行为日志不再是后台沉默的数据表，而成为驱动各类业务功能的统一基础。

### 1.2 国内外研究现状

国内关于电商用户行为分析的研究大致沿着三条路径展开。第一条路径聚焦行为日志挖掘与消费预测，通过浏览日志、查询日志或购买日志识别用户兴趣迁移与转化规律。王菁等围绕日志挖掘提出面向电商场景的查询建议方法，说明日志序列能够支持更加精准的行为理解[1]。张鹏翼等利用用户浏览日志开展移动购买预测研究，表明日志数据具备较强的消费意图识别价值[2]。丁鹏程从系统实现角度讨论了电商平台用户行为分析系统的功能组织方式，说明将行为采集、分析与展示放在同一平台中具有较好的应用价值[3]。第二条路径聚焦用户价值分析与分层，RFM 及其改进模型在电商、零售、农业电商和营销优化场景中应用广泛。刘涛等基于改进 RFM 模型进行用户需求响应潜力分级，体现了多维价值指标在用户识别中的适用性[4]。陈海燕等将改进 RFM 与 K-means 结合用于淘宝用户行为分析，说明结构化分层在营销定位中具有较强的实用性[5]。杨晓梅、胡志仁等的研究则从客户维系和农产品电商运营角度，进一步说明了 RFM 思路在电商客户管理中的可迁移性[6][7]。第三条路径聚焦可视化与系统实现，一些研究开始关注如何使用 Python、ECharts、Vue 等技术构建分析平台，使统计结果能够被非技术人员直接理解和使用。李宏鹏等使用 Python 与 ECharts 设计农业气象数据可视化平台，表明轻量级技术栈也可以构建较完整的数据展示系统[8]。夏祎忞对 Vue 组件化开发模式进行了实践分析，为前端模块组织提供了参考[9]。管玲玲等对 Flask 前后端分离 Web 开发实现进行了总结，为本文的技术路线提供了直接借鉴[10]。

国内研究已经为本文提供了扎实基础，但也存在两个较明显的不足。其一，很多研究把“分析方法”与“系统实现”分开论述，算法部分强调模型效果，系统部分强调界面或数据库设计，二者之间缺少贯通。其二，部分系统在写作时采用了较多复杂术语，例如实时采集、智能决策、精准营销，但在功能实现上并未建立清晰的证据链，导致分析结果缺乏说明性。对于毕业设计来说，这种写法很容易造成“概念很大、系统很轻”的问题。

国外研究更早关注日志数据的建模与行为识别。Xie 等基于搜索引擎日志挖掘用户行为特征，体现出日志数据在用户研究中的基础价值[11]。Borowiec 等使用 Web 应用日志数据集结合数据挖掘技术进行用户行为识别，说明日志分析已经不仅服务于统计展示，也服务于行为模式识别[12]。Tingzhong W 等围绕电商用户行为日志构建可视化分析方法，强调日志分析结果必须通过图形化界面转化为可理解的信息[13]。Cao 等将协同推荐算法引入电商行为分析，体现了日志、推荐和运营决策之间的耦合关系[14]。Yuan 等围绕电商用户行为数据分析技术展开研究，说明行为数据分析正在从单一统计走向更综合的经营应用[15]。近年的研究还进一步关注冷启动和用户画像问题，Ginidi 等通过合成用户画像与深度学习缓解推荐系统中的冷启动问题，为用户画像构建和推荐回退提供了新的思路[16]。

国外研究的优势在于方法体系较完整，尤其重视行为数据与推荐、画像、识别之间的结合。但国外研究大多建立在较成熟的数据基础设施之上，强调大规模分布式处理、复杂模型训练和多源数据融合，这与本科毕业设计所面向的中小规模、本地化演示环境并不完全一致。因此，本文不照搬复杂工业方案，而是在吸收用户画像、日志分析、可视化展示和推荐回退思想的基础上，构建一套更适合课程设计与毕业答辩场景的系统实现路线。

总体来看，现有研究已经证明用户行为日志能够支撑用户画像、经营分析、推荐计算和可视化决策，但在“多角色统一闭环、分析结果可解释、系统演示可信”这三个方面仍存在整合空间。本文的工作重点就在于将这些环节组织成一个结构完整、逻辑清楚、演示稳定的系统。

### 1.3 研究意义

本研究的意义可以从理论和实践两个层面展开。

从理论层面看，用户行为日志本身并不是新概念，但如何在一个完整系统中将其用于推荐、画像、经营分析和结果解释，仍然值得系统化梳理。本文将用户行为日志作为统一的数据底座，以行为类型、类目偏好、活跃时间、价格区间和商品热度为分析入口，在系统层面将日志记录、行为建模、可视化分析和角色展示联系起来。这样的组织方式能够将分散的功能点转化为可验证的分析链路，为轻量级电商分析系统的设计提供一套可参考的实现思路。

从实践层面看，毕业设计项目常见的问题不是功能数量不足，而是功能之间缺乏关联。用户侧只显示推荐，商家侧只显示统计，管理员侧只显示任务，三者互不解释，容易造成页面很多但逻辑割裂。本文构建的系统强调三类角色共享同一行为日志基础。用户产生行为后，画像和推荐随之变化；商家侧能够看到热点、冷门和活跃信号；管理员侧能够看到日志输入、聚合规则、评分机制和解释结果。这种设计更符合答辩中对“系统完整性”和“结果可信度”的要求。

此外，本文在方法选型上坚持适度原则。对于毕业设计来说，一味引入复杂深度学习模型并不一定比规则化、可解释的方法更合适。系统采用行为权重评分、时间窗口比较、偏好类目提取、回退推荐和简化分层等方法，虽然相对朴素，但每一步都能够说明依据，结果也能通过界面展示与测试验证。这样的方案更适合教学场景，也更容易被老师认可。

### 1.4 研究内容

本文围绕“基于用户行为日志的电商智能分析与可视化系统”这一主题，主要开展以下工作。

第一，完成系统总体方案设计。根据管理员、商家和普通用户三类角色的实际需求，设计系统的总体架构、接口组织方式、数据结构和功能模块边界，形成前后端分离的实现方案。

第二，构建用户行为日志处理闭环。系统既支持管理员批量模拟生成行为日志，也支持普通用户在推荐页面通过浏览、收藏、加购、购买等真实操作写入个人行为记录。通过这两类日志来源，形成兼顾演示数据规模和真实交互反馈的日志基础。

第三，完成关键分析模块实现。系统围绕日志数据实现商品热度计算、冷热门商品识别、地区和类目热点统计、用户画像构建、简化用户分层、商家异常预警以及基于偏好与热度的推荐输出，并通过角色页面进行有针对性的展示。

第四，构建可视化与解释层。商家端以经营驾驶舱方式展示总览、核心业务、辅助分析和详细分析内容；管理员端提供日志预览和算法处理链路页面；用户端展示推荐结果、画像依据和近期行为记录。通过统一的界面组织方式，把后台分析过程转换成能够直接展示给老师和使用者的结果。

第五，对系统进行测试验证。通过后端单元测试、前端界面测试和系统级功能检查，验证核心业务是否稳定运行，并结合页面结果分析系统的完整性和可信度。

## 2 相关技术与理论基础

### 2.1 系统开发技术

系统前端采用 Vue 3 作为核心开发框架。Vue 3 使用组合式 API 组织页面状态和交互逻辑，适合构建多页面、多角色的前端应用。与传统模板式开发方式相比，Vue 3 在组件复用、状态组织和页面拆分方面更灵活，能够较好地支撑商家看板、管理员控制台和用户推荐页等复杂界面。前端工程化工具选择 Vite。Vite 启动速度快，模块热更新效率高，适合毕业设计阶段的高频调试。状态管理采用 Pinia，用于维护用户认证信息、角色信息和部分页面共享状态。路由层使用 Vue Router，承担登录跳转、角色页面隔离和商家侧模块切换等功能。图表可视化基于 ECharts 实现，用于呈现漏斗、地区热点、类目分布、趋势和结构化统计结果。

后端采用 Flask 框架构建 RESTful 接口服务。Flask 本身轻量、易扩展，适合毕业设计项目快速搭建和按模块组织业务逻辑。系统结合 Flask-SQLAlchemy 管理数据模型和数据库访问，以对象关系映射方式组织用户、商品、任务等结构化业务数据。用户认证采用 Flask-JWT-Extended 实现，通过 JWT 完成登录后的身份校验和角色访问控制。与基于服务端 Session 的方案相比，JWT 机制更适合前后端分离场景，能够降低接口耦合度，并使管理员、商家和用户三类角色的权限验证保持统一。

系统数据存储采用 MySQL。系统通过 Flask-SQLAlchemy 统一管理用户、商品、任务等结构化业务数据与行为日志数据，便于在答辩演示中直接展示“日志表真实存在、数据可查询”的效果。行为日志以事件为单位持久化到 `behavior_logs` 表中，既支持管理员侧批量模拟生成，也支持用户侧真实操作写入，为商家经营分析、用户画像与推荐提供统一的数据底座。为避免日志长期累积导致数据膨胀，系统提供“保留最新 N 条”的手动清理能力，用于演示环境下的可控维护。

### 2.2 用户行为日志建模基础

用户行为日志是系统分析能力的核心数据来源。本文中的行为日志以单条事件为基本单位，每条记录主要包含日志标识、用户标识、商品标识、商家标识、商品类目、行为类型、价格、地区、时间戳和来源渠道等信息。行为类型主要包括浏览、点击、收藏、加购和购买。相比仅记录订单信息的做法，行为日志能够更加完整地反映用户从兴趣产生到最终成交的全过程。

为了兼顾演示数据规模和用户真实反馈，本文采用两类日志来源。一类是系统模拟生成的行为日志。管理员可通过任务管理模块快速生成大量行为数据，用于构造经营看板、趋势图表和分析结果；日志预览模块则用于分页查看落库日志明细。另一类是用户前台真实操作日志。普通用户在推荐页对商品执行浏览、收藏、加购和购买后，系统会即时写入对应行为。真实操作日志主要用于用户画像、个性化推荐和近期行为反馈，避免页面展示出现“用户没有操作却被说成有强偏好”的不合理情况。

在日志建模时，本文特别强调“来源渠道”的区分。系统为真实前台行为设置专门的 source_channel 标识，使推荐和画像逻辑只识别来自用户页面的真实行为，而不会把模拟日志误判为当前用户的个性化依据。这一处理虽然实现简单，却对系统可信度非常关键。它保证了用户侧展示的画像和推荐更符合实际交互，也使答辩中关于“这些画像依据从哪里来”的问题能够被直接说明。

### 2.3 关键分析方法

在分析方法设计上，本文坚持“可解释、可落地、与系统规模匹配”的原则，没有盲目引入训练成本高、解释难度大的复杂模型，而是采用规则化与统计式方法支撑主要功能。对本科毕业设计而言，算法的价值并不只体现在模型复杂度上，更体现在是否与数据规模匹配、是否能说明输入输出关系、是否能经受答辩追问。基于这一考虑，本文把算法设计重点放在行为权重、偏好提取、回退推荐和时间窗口比较上。

商品热度计算采用行为权重评分模型。系统将不同类型的用户行为赋予不同权重，其中浏览记为 1 分，点击记为 2 分，收藏记为 3 分，加购记为 5 分，购买记为 8 分。这样做的原因在于，不同行为体现的用户意图强度不同。单纯浏览并不一定代表强需求，而购买行为通常意味着较强转化意图。系统通过对日志中各类行为进行加权累积，得到商品的热度分值，据此输出热销商品与冷门商品列表。该方法具有实现简单、解释清楚、结果直观的特点，也便于在管理员算法链路页面中直接展示。若以商品 \(p\) 为对象，其热度值可表示为：

\[
HotScore(p)=1\times View(p)+2\times Click(p)+3\times Favorite(p)+5\times Cart(p)+8\times Purchase(p)
\]

该模型虽然不是复杂预测模型，但对毕业设计场景十分适合。一方面，权重设置与行为意图强弱相匹配；另一方面，热度变化能够通过日志明细直接复核，便于老师检查系统是否“算得出来”。

用户画像构建采用行为统计与偏好提取相结合的方法。系统从真实前台行为中统计用户高频交互的商品类目、行为类型、活跃时间段和价格区间，并形成近期行为摘要和画像标签。价格偏好并不简单根据所有商品均值计算，而是优先参考购买行为，再参考加购和收藏行为，从而使结果更接近用户的真实消费倾向。对于没有足够行为数据的新用户，系统输出空画像结构并标记为冷启动状态。

推荐模块采用“偏好类目 + 热度排序 + 回退机制”的混合推荐思路。对已有真实行为的用户，系统先从用户近期高频行为中提取偏好类目，再在可售商品中优先筛选相关候选，并结合热度分进行排序；对行为数据不足的冷启动用户，系统直接采用全局热度回退推荐。该方法避免了在小规模数据场景下使用复杂协同过滤所带来的稀疏性问题，同时保留了个性化倾向与结果可解释性。林廷昱关于 RFM 与协同过滤结合的研究说明，用户价值与推荐候选之间可以形成联动关系[17]。本文没有直接复刻复杂协同过滤流程，而是把“偏好提取”与“热点回退”组合起来，在小样本环境下获得更稳定的效果。

异常预警采用时间窗口对比的规则检测方法。系统围绕最近一段时间内的行为数据，从流量上升但转化偏低、收藏较高但加购较低、地区流量异常波动等角度识别潜在问题。相比使用黑箱式预测模型，规则检测方法更适合毕业设计场景中的异常展示，因为每一类异常都可以追溯到清晰的指标定义和时间窗口比较逻辑。具体实现中，系统将最近两天与此前三天作为两个比较窗口，分别计算浏览、收藏、加购和购买等指标的变化情况。当近期浏览显著上升但购买比率没有同步提升时，系统输出“流量增长但转化偏低”提示；当收藏量较高而加购响应不足时，系统输出“兴趣较高但转化链路承接不足”提示。该逻辑不依赖训练样本，结果解释直接，适合商家页面展示。

用户分层采用简化分层思路。本文参考 RFM 方法中的最近活跃、购买频次和消费金额三个维度，但没有把其设计为标准化评分模型，而是根据系统规模与数据量使用简化规则进行分层。这样既保留了“价值用户识别”的分析意义，也避免在数据基础有限的情况下过度包装算法能力。简化分层的基本思路是：最近活跃天数较短、购买频次较高且累计消费金额较高的用户归入高价值群体；最近活跃稳定、购买频次中等的用户归入潜力群体；近期几乎没有购买且活跃明显下降的用户归入待激活群体。该方法吸收了 RFM 的思想，但刻意避免把系统包装为复杂评分引擎，更符合当前数据量级和论文写作的严谨性。

## 3 系统需求分析

### 3.1 业务需求分析

系统面向三类角色开展设计。管理员的核心关注点在于系统演示控制、行为日志预览、批量模拟任务和算法链路说明。管理员并不直接面向商品经营，而是从全局角度确保系统的数据输入、任务运行和分析过程可展示、可解释。商家的核心关注点在于经营结果。商家希望快速看到当前商品热度、地区和类目热点、近期活跃信号、异常预警以及用户价值结构，从而判断商品运营状态。普通用户的核心关注点则是个性化体验，包括推荐是否合理、画像是否可信、近期操作是否被系统及时识别。

结合上述角色特征，系统需要满足三个基本业务目标。第一，系统要能稳定构造并维护行为日志数据，使各页面拥有共同的数据基础。第二，系统要能根据不同角色输出不同维度的分析结果，而不是让所有页面简单重复同一批统计数据。第三，系统要能解释分析结果的来源，尤其要避免推荐、画像和经营结论与实际数据相脱节。

### 3.2 功能需求分析

从功能层面看，系统主要包含用户认证、日志生成与记录、商家经营分析、用户画像与推荐、管理员日志管理与链路展示五个核心模块。

用户认证模块负责系统登录、身份校验和角色访问控制。系统需要支持管理员、商家和普通用户登录，并根据角色展示不同导航和页面内容。该模块不仅影响安全性，也决定了多角色业务是否能够被清楚地区分。

日志生成与记录模块负责行为数据来源的组织。一方面，管理员需要能够快速生成一定规模的模拟日志，用于演示分析链路和看板结果。另一方面，普通用户在推荐页的真实行为应被系统记录，用于影响后续画像和推荐结果。二者共同构成系统的行为日志基础。

商家经营分析模块是系统的核心展示区。它需要覆盖经营总览、事件级转化指标、地区热点、类目热点、热销商品、冷门商品、近期用户活跃信号、简化用户分层和异常预警等内容。商家不仅需要看到统计数字，更需要通过图表和摘要快速理解经营状态。

用户画像与推荐模块负责将日志分析结果返回给普通用户。推荐页面需要给出推荐商品、推荐理由和冷启动说明；画像页面需要展示偏好类目、活跃行为、价格倾向和近期依据；近期行为面板需要帮助用户理解自己刚刚的操作如何影响系统结果。

管理员日志管理与链路展示模块用于回答“日志如何被维护、系统如何处理日志”的问题。该模块既要支持管理员完成批量模拟生成、日志分页预览与手动清理（保留最新 N 条），也要把原始日志输入、聚合方式、权重评分、画像与推荐、异常检测、分层结果以及基于可配置大模型的解释性摘要串联起来，使系统不只是给出结果，还能展示分析过程。

### 3.3 非功能需求分析

系统需要满足基本的性能、可用性、安全性和可维护性要求。性能方面，本课题并不追求工业级高并发能力，但要求在本地演示环境下能够较流畅地完成接口响应、图表渲染和日志生成。可用性方面，系统应具有清晰的导航结构、明确的文案说明和稳定的页面切换流程，避免用户在操作过程中迷失。安全性方面，系统需要对不同角色进行后端权限校验，避免用户越权访问不属于本角色的数据。可维护性方面，系统应使用相对清晰的前后端分层方式组织代码，使后续功能扩展和问题修复具有较好的可操作性。

此外，由于本系统面向毕业设计答辩场景，系统还具有一种特殊的非功能要求，即展示可信度。页面所使用的术语必须与真实实现保持一致。例如，系统采用事件级行为统计，就不能把结果说成完整路径漏斗；系统采用简化分层，就不能直接包装成复杂 RFM 模型；用户没有真实行为时，页面要明确显示冷启动，而不是虚构个性化结果。这类要求虽然不直接属于传统的软件性能指标，但对毕业设计质量影响非常大。

## 4 系统总体设计

### 4.1 设计目标

系统设计的首要目标是构建一条完整、清晰、可验证的用户行为日志分析闭环。具体来说，系统要实现三点。第一，数据层面能够形成从基础数据到行为日志再到分析结果的关联。第二，业务层面能够支撑管理员、商家和普通用户三类角色的差异化需求。第三，展示层面能够把分析结果以图表、列表和解释说明的形式呈现出来，让使用者和答辩老师都能理解系统在做什么。

在设计原则上，本文重点遵循模块清晰、口径统一、技术适配和结果可解释四项原则。模块清晰意味着前后端职责划分明确，各分析功能围绕独立接口和独立页面组织。口径统一意味着同一指标在不同页面中的定义保持一致，避免商家页、管理员页和用户页各说各话。技术适配意味着所有技术选型都服务于当前系统规模与实现目标，不盲目引入超出需求的复杂组件。结果可解释意味着推荐、画像、热度、异常和分层都必须说明依据，而不是只给出结论。

### 4.2 总体架构设计

系统整体采用前后端分离架构。前端为基于 Vue 3 的单页面应用，负责登录交互、路由切换、角色页面渲染和图表展示。后端为基于 Flask 的 RESTful API 服务，负责用户认证、业务计算、日志生成和数据聚合。前后端通过 JSON 数据结构进行通信。该架构一方面能够提高页面交互的灵活性，另一方面也便于后续对接口和界面分别迭代。

从逻辑层次看，系统可以分为表现层、接口层、业务服务层和数据层。表现层包含管理员端、商家端和用户端页面。接口层对外提供统一的分析、推荐、认证、管理等接口。业务服务层封装日志模拟、画像构建、推荐排序、统计分析、异常检测等功能。数据层以 MySQL 为统一存储，包含用户、商品、任务等基础数据与 `behavior_logs` 行为日志表。这样组织后，各层职责较为清楚，既能保障代码可维护性，也有助于论文中的系统说明。

### 4.3 数据结构设计

系统中的结构化数据主要包括用户、商品和任务三类实体。用户实体包含用户名、密码摘要、角色等信息，用于实现登录认证和角色管理。商品实体包含商品名称、价格、库存、是否上架、所属类目和商家标识等字段，是推荐、热销统计和价格倾向分析的核心基础。任务实体用于管理员侧模拟任务管理，记录任务状态、批次信息和执行情况。

行为日志在系统中持久化到 MySQL 的 `behavior_logs` 表中，并在逻辑上保持清晰的数据结构。每条日志记录用户、商品、商家、类目、行为类型、价格、地区、时间、来源渠道等字段，用于支撑商家经营分析、用户画像与推荐逻辑。由于系统需要区分模拟日志与真实行为日志，来源渠道字段具有较强的业务意义。通过这一字段，推荐模块可以仅基于用户侧真实行为进行个性化计算，避免把模拟日志误判为个体偏好依据。

### 4.4 功能模块设计

用户认证模块设计上采用“登录接口 + JWT 鉴权 + 路由守卫”的组合方式。后端负责核验用户身份并下发令牌，前端负责存储令牌并基于角色控制页面入口。这样可以保证角色隔离在接口层和页面层都得到落实。前端登录完成后会把令牌和用户基础信息写入状态仓库，再通过路由守卫判断目标页面是否与当前角色匹配。后端接口则在进入具体业务逻辑前统一校验令牌与角色，从而形成双重保护。

日志生成与记录模块分为后台模拟和前台记录两条链路。后台模拟链路由管理员触发，能够快速形成较大规模行为日志，用于支撑统计图表和经营分析。前台记录链路由普通用户发起，操作结果会立即写入个人行为日志，并影响推荐和画像结果。通过双链路设计，系统既具备一定的数据规模，也保留真实交互反馈。

商家经营分析模块围绕经营总览、热点分析、事件级转化指标、冷热门商品、近期活跃信号、简化用户分层和异常预警展开。各模块由独立接口提供数据，前端以统一的看板方式组织内容。用户画像与推荐模块则更强调个体视角，突出推荐原因、偏好依据和近期行为变化。管理员链路模块则面向解释与展示，把系统如何从日志得到结果的过程显式展示出来。

## 5 系统实现

### 5.1 用户认证与权限控制实现

系统登录流程由前端登录页、后端认证接口和令牌校验机制共同完成。用户提交用户名和密码后，后端首先验证账号是否存在以及密码是否匹配，验证通过后生成 JWT 令牌，并将用户标识与角色信息写入令牌载荷。前端在登录成功后保存令牌与角色信息，根据不同角色跳转到相应首页。管理员进入系统总览和任务管理页面，商家进入经营驾驶舱，普通用户进入推荐首页。

权限控制采取“后端为主，前端为辅”的实现策略。前端根据角色控制导航菜单和页面入口，主要用于提升交互体验；真正的数据权限控制仍然由后端接口完成。例如，商家经营分析接口只允许商家角色调用，普通用户推荐行为记录接口只允许普通用户访问，管理员算法链路接口只允许管理员读取。这样的设计能避免前端绕过限制直接访问不属于本角色的数据。

在具体实现中，系统把认证逻辑与业务接口解耦。认证模块只负责登录、令牌解析和基础角色识别，推荐、分析、管理等接口则根据 JWT 中的角色和身份信息进行进一步判断。这种组织方式使代码结构更清晰，也方便论文中说明系统的安全边界。

### 5.2 日志生成与行为记录实现

为了让系统能够在答辩环境中快速展示分析结果，本文引入了日志模拟机制。管理员在任务管理页面触发批量行为日志生成。系统根据已有用户、商品和类目数据，按预设规则生成浏览、点击、收藏、加购和购买等事件，并批量写入 MySQL 的 `behavior_logs` 表。该机制能够在短时间内形成具有一定规模和结构差异的行为数据，用于后续商家分析和可视化展示。模拟规则综合考虑商品所属类目、用户所在地区、时间分布和行为权重，保证生成的日志并非完全随机堆叠，而是具备一定统计特征。

单纯依赖模拟日志会带来一个问题：用户侧推荐和画像可能失真。为了解决这一问题，系统在用户推荐页面中增加了真实行为记录接口。用户对推荐商品执行浏览、收藏、加购或购买操作后，后端会即时写入新的行为记录，并标记来源渠道为用户前台页面。推荐服务和画像服务只读取这一来源下的个人日志，因此用户看到的推荐和画像能够与自己的实际操作保持一致。

在实现过程中，系统还增加了可售商品过滤逻辑。推荐接口只从上架且库存大于零的商品中选择候选，用户行为记录接口也会拒绝对不可售商品写入操作日志。这一改动看似细小，但对系统可信度十分重要。因为如果下架商品仍然出现在推荐中，或者无库存商品还能“购买”，整个系统的业务合理性就会受到影响。

### 5.3 商家经营分析实现

商家经营分析模块是系统页面最复杂的部分。为了避免传统看板页面信息堆叠过度，系统将商家端组织为经营总览、核心业务、辅助分析和详细分析四个板块，并通过左侧固定导航实现单页切换。经营总览模块用于展示关键指标、核心摘要和经营简报；核心业务模块用于展示热销商品、冷门商品、近期活跃信号和地区类目重点信息；辅助分析模块用于展示简化用户分层和关联分析；详细分析模块用于展示事件级转化指标、异常预警和更完整的明细内容。页面切换不再采用传统长页面锚点跳转，而是采用基于路由参数的单面板切换方式。这样做可以使商家在答辩演示时快速聚焦某一板块，避免一个页面同时堆叠过多内容。

经营总览中的关键指标并非简单统计数据库记录数，而是围绕商家自身日志计算行为总量、浏览行为数、购买行为数和购买率。购买率的定义为购买行为次数与浏览行为次数之比，这种口径在系统中被统一用于商家看板和 AI 解释输入，避免不同模块对同一指标使用不同定义。

热销商品与冷门商品识别基于行为权重评分模型完成。系统将浏览、点击、收藏、加购、购买分别赋予不同权重，通过累积各商品的行为得分得到热度排序，得分高的商品进入热销列表，得分较低且长期缺乏高意向行为的商品进入冷门列表。地区热点和类目热点则基于日志中的地区和类目信息进行计数与排序。这样的实现方式虽然不是复杂机器学习模型，但足够符合经营分析场景，且具有较好的可解释性。

近期用户活跃信号模块不再使用“变化”这一容易引起误解的表述，而是明确展示最近二十四小时窗口内的收藏、加购和购买等高意向行为信号。异常预警模块则通过时间窗口比较识别流量异常、转化偏低和地区波动等情况。用户分层模块采用简化分层方式，根据最近活跃、购买频次和金额结构识别高价值、潜力和待激活用户。这些模块共同构成商家端的核心经营分析能力。

### 5.4 用户画像与推荐实现

用户侧实现主要围绕两个问题展开：推荐结果是否合理，画像依据是否可信。系统在推荐逻辑中采用“偏好类目 + 热度排序 + 冷启动回退”的混合思路。对于已经产生真实前台行为的用户，系统首先统计用户高频交互类目，再结合全局热度分在可售商品池中筛选候选，输出偏向个性化的推荐结果。对于缺少真实行为的新用户，系统直接采用热度回退推荐，并在页面上明确标记当前属于冷启动状态。

推荐结果不仅需要给出商品，还需要说明依据。系统在页面上展示推荐模式、推荐理由以及近期行为记录，使用户能够看到“自己刚刚的哪些操作影响了推荐”。这一设计的意义在于减少系统的神秘感，让推荐结果更容易被理解。对毕业设计答辩而言，这种解释性同样重要，因为老师往往会追问推荐依据和数据来源。系统还为每件推荐商品提供四类操作按钮，分别对应浏览、收藏、加购和购买。用户触发行为后，系统会重新拉取推荐结果与画像信息，形成即时反馈。这样的实现虽然增加了前后端交互次数，但明显增强了系统的演示效果。

画像模块主要展示用户偏好类目、常见行为、价格倾向、近期行为依据和画像标签。价格倾向优先依据购买行为生成，当购买数据不足时再参考加购和收藏行为，以提高结果合理性。页面还专门保留“最近五条画像依据”区域，用于显示画像所依赖的个人行为日志，而不再把系统模拟日志混入用户个体结论。通过这一调整，用户侧画像和推荐在业务逻辑上更加稳健。

### 5.5 管理员日志管理与链路展示实现

管理员端的设计重点不是传统后台管理，而是围绕演示与解释需求，把“日志如何进入系统、如何被处理、如何被维护”这件事讲清楚。为此，管理员端按职责拆分为任务管理、日志预览与处理链路展示三类能力。

在任务管理页面中，管理员可手动触发批量模拟生成行为日志，并可执行“保留最新 N 条”的日志清理操作。该功能一方面用于快速构造演示数据规模，另一方面用于控制日志持续增长带来的维护成本，使系统在反复演示与测试过程中保持稳定。

在日志预览页面中，系统以分页方式从 `behavior_logs` 表读取日志明细，帮助使用者直观看到日志字段长什么样、数据是否真实落库以及不同来源渠道的差异。该页面专注于日志查看，不再混入生成与清理入口，避免页面职责混乱。

在处理链路展示页面中，系统把原始日志输入、聚合统计、行为权重评分、画像与推荐、异常检测、简化用户分层以及基于可配置大模型的解释性摘要等节点按顺序展示出来。每个节点不仅显示数量与关键指标，还配有简短说明，帮助使用者理解这一阶段做了什么，从而增强系统可解释性与答辩说服力。

## 6 系统测试与结果分析

### 6.1 测试环境与测试方法

系统测试环境采用 Windows 本地开发环境。后端使用 Python 运行 Flask 服务，并通过 pytest 对核心接口和分析逻辑进行验证。前端使用 Node.js 环境运行 Vue 3 工程，通过 Vitest 对页面逻辑和渲染结果进行测试。数据库采用 MySQL（可通过环境变量配置连接信息），便于在测试与演示中直接核对行为日志表与数据写入情况。由于本文的目标是构建一套稳定可演示的毕业设计系统，测试重点放在功能正确性、角色闭环完整性和关键页面展示可信度上。

测试方法主要包括三类。第一类是后端接口与逻辑测试，重点验证认证、推荐、分析、异常与日志处理等核心服务是否按预期运行。第二类是前端页面测试，重点验证商家页、管理员页和用户页在不同数据输入下是否能正确展示结果。第三类是人工场景测试，模拟管理员、商家和普通用户三类角色的典型使用路径，检查页面切换、行为记录、数据联动和提示文案是否合理。

### 6.2 功能测试

在用户认证方面，系统对三类角色登录、页面跳转和权限访问进行了验证。测试表明，管理员、商家和普通用户均能按角色进入对应界面，不同角色之间的菜单和接口访问边界清晰，未出现明显的越权访问问题。

在推荐与画像方面，测试重点放在冷启动状态、真实行为写入、个性化推荐切换和画像依据显示。测试结果表明，当用户尚未产生真实行为时，系统能够以冷启动模式返回回退推荐；当用户在推荐页面执行浏览、收藏、加购和购买后，近期行为面板、画像依据和推荐结果均会随之更新。后端测试同时验证了不可售商品不会进入推荐结果，不可售商品也无法写入推荐行为。

在商家经营分析方面，测试重点放在商家作用域统计是否正确、关键指标口径是否统一、页面文案是否与实现一致。系统已对商家总览、热点商品、冷门商品、地区类目分布、近期活跃信号、简化用户分层和异常预警进行验证。测试结果表明，各模块能够围绕当前商家数据进行展示，未再出现将全局日志直接混入商家视角的问题。

在管理员链路展示方面，测试重点放在算法处理节点是否能正常加载，以及说明文案是否准确。测试结果表明，日志输入、聚合统计、评分机制、画像推荐、异常检测、用户分层和 AI 解释等节点均可成功展示，页面说明也能够与当前实现保持一致。

### 6.3 测试结果分析

系统当前已完成后端与前端的多轮回归测试。后端共 66 个测试用例，覆盖推荐逻辑、画像、异常和商家分析等重要功能；前端共 38 个测试用例，覆盖商家页、管理员页、用户首页和布局逻辑。测试结果显示，系统能够稳定完成三类角色的业务闭环，核心功能运行正常，关键页面渲染符合预期。

从结果分析看，本文系统的优势主要体现在三个方面。第一，业务闭环清楚。管理员可以生成并查看日志，商家可以依据日志分析经营状态，用户可以通过真实行为影响自己的推荐和画像，三者之间形成了清晰联系。第二，分析结果可解释。热度、画像、推荐、异常和分层均基于明确规则生成，不存在无法说明来源的“黑箱式结果”。第三，系统演示可信度较高。近期对推荐可售过滤、事件级指标口径和简化分层文案进行了修正后，页面展示与真实实现的一致性更好。

当然，系统仍然存在一定局限。一方面，日志数据主要依赖模拟生成和本地环境运行，尚未接入真实线上埋点和大规模流处理链路。另一方面，推荐、异常和分层方法更强调可解释性和答辩适配性，尚未引入更复杂的机器学习模型。对于毕业设计而言，这样的取舍是合理的；对于后续扩展而言，也预留了进一步提升的空间。

## 7 总结与展望

本文围绕“基于用户行为日志的电商智能分析与可视化系统设计与实现”这一主题，完成了一套面向毕业设计场景的多角色系统设计与开发工作。系统以用户行为日志为核心数据基础，采用 Flask、Vue 3、Vite、Pinia、ECharts、MySQL 和 JWT 等技术，实现了用户认证、日志生成与记录、商家经营分析、用户画像与推荐、管理员日志管理与链路展示等核心功能。系统在方法层面采用行为权重评分、偏好类目提取、热度排序、时间窗口对比和简化分层等可解释方案，并通过行为日志落库、分页预览、手动清理与证据化推荐理由等机制提升演示可信度，使功能实现与展示结果之间能够形成清晰对应关系。

从最终效果看，本文的工作并不在于构造一个追求大规模数据处理能力的工业级平台，而在于构建一条结构完整、逻辑清晰、结果可信的行为日志分析链路。系统能够让管理员说明“日志如何被处理”，让商家看到“经营状态如何被分析”，也让普通用户理解“推荐为什么这样出现”。这使得整套系统既具备软件工程意义，也具备较好的毕业设计答辩展示价值。

后续工作可从三个方向继续展开。第一，可以引入真实埋点、消息队列和持久化日志存储，提高系统对真实业务环境的适配能力。第二，可以在数据规模扩大后尝试引入协同过滤、序列推荐或更细粒度的异常检测模型，提升分析精度。第三，可以继续优化前端可视化表现和导出能力，使系统不仅适用于答辩展示，也能适用于更真实的运营辅助场景。
