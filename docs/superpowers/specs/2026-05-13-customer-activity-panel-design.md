# 用户端双层最近行为面板设计

## 1. 目标

本次增强用于补齐用户端“行为证据展示”能力，让老师不仅能看到用户点击了浏览、收藏、加购、购买等动作，还能立刻在页面里看到这些动作被系统记录并用于支撑画像分析。

当前系统已经具备：

- 用户行为写日志
- 推荐结果刷新
- 画像结果刷新

但页面仍然缺少一个直观的“行为记录展示区”。这会导致老师虽然看到推荐和画像变化，却不一定能直接看见“刚刚到底记录了什么行为、系统依据的近期行为又是什么”。

因此本次目标是把用户行为展示拆成两层：

1. `本次会话刚刚操作`
2. `最近 5 条个人行为`

前者用于强调答辩现场操作，后者用于强调画像和推荐的分析依据。

## 2. 设计范围

本次只做用户首页中的行为展示增强，不新增新页面。

本次范围包括：

- 在用户首页增加 `本次会话刚刚操作` 面板
- 在用户首页增加 `最近 5 条个人行为` 面板
- 会话内动作在前端本地即时展示
- 历史最近行为继续复用画像接口中的 `recent_activity`

本次不做：

- 独立行为历史页
- 行为分页
- 行为搜索和筛选
- 服务端会话状态存储
- 跨页面、跨刷新保留“本次会话刚刚操作”

目标是用最小改动，把“刚刚做了什么”和“系统最近依据了什么”同时展现出来。

## 3. 核心方案

### 3.1 双层结构

用户首页新增两个行为展示区：

#### 第一层：本次会话刚刚操作

该区域强调“你刚刚点过什么”，主要用于答辩演示。

展示内容包括：

- 商品名称
- 行为类型
- 简短时间标记，例如 `刚刚`

数据来源：

- 前端本地状态

更新时机：

- 用户点击任意行为按钮成功后，立即插入一条会话内行为记录

#### 第二层：最近 5 条个人行为

该区域强调“系统最近依据了哪些个人行为来形成画像”。

展示内容包括：

- 商品名称
- 行为类型
- 时间

数据来源：

- 继续复用 `GET /api/users/portrait` 中已有的 `recent_activity`

更新时机：

- 每次刷新用户首页数据后同步更新

### 3.2 为什么要分成两层

如果把“刚刚操作”和“历史行为”混在一个列表中，老师不容易分辨：

- 哪些是你现场点的
- 哪些是系统此前已经记录的历史行为

双层结构能同时解决两个问题：

- `演示性`：老师可以直观看到你刚刚做了什么
- `可信度`：老师可以直观看到画像和推荐不是空口推断，而是有最近行为作为依据

## 4. 前端设计

### 4.1 本次会话刚刚操作

前端新增一个本地状态列表，例如：

- `sessionActions`

列表元素结构建议为：

```json
{
  "product_id": 1,
  "product_name": "轻量跑鞋",
  "action_type": "favorite",
  "action_label": "收藏",
  "time_label": "刚刚"
}
```

行为按钮点击成功后：

1. 调用后端写日志接口成功
2. 将本条记录插入 `sessionActions` 顶部
3. 保留最近若干条，例如最近 5 条
4. 再执行原有刷新逻辑拉取推荐、趋势、画像

这样可以保证“刚刚操作”即时可见，而不依赖后端重新返回一套专门的会话数据。

### 4.2 最近 5 条个人行为

该区域直接使用当前画像结果中的：

- `profile.recent_activity`

前端只负责把它渲染成列表，不新增额外请求。

展示格式建议为：

- 商品名
- 行为标签
- 时间

例如：

- `轻量跑鞋 · 收藏 · 2026-05-13 10:20`

### 4.3 页面布局建议

为了避免用户首页信息过挤，两个行为区采用“一个轻量卡片 + 一个依据卡片”的方式：

1. `本次会话刚刚操作`
   - 放在推荐卡片附近
   - 让老师先看到操作反馈

2. `最近 5 条个人行为`
   - 放在画像卡片附近
   - 让老师自然理解画像依据

这样页面信息流是顺的：

- 我刚刚做了什么
- 系统因此怎样理解我
- 系统接着给我什么推荐

### 4.4 空状态设计

#### 本次会话刚刚操作为空时

显示：

- `你还没有进行操作，可先点击浏览、收藏、加购或购买。`

#### 最近 5 条个人行为为空时

冷启动阶段显示：

- `当前暂无历史个人行为记录。`

这样既不会显得页面坏掉，也能和冷启动逻辑保持一致。

## 5. 后端设计

本次原则上不新增接口，也不改画像接口结构。

原因是：

- `本次会话刚刚操作` 由前端本地状态就能完成
- `最近 5 条个人行为` 已经能通过画像接口拿到

因此本次后端职责不变：

- 继续通过行为写日志接口记录用户行为
- 继续通过画像接口返回 `recent_activity`

如果后续需要跨刷新保留“本次会话刚刚操作”，再考虑增加服务端会话存储或独立行为查询接口。但这不在本次范围内。

## 6. 数据流

用户点击推荐卡片按钮后的完整流程如下：

1. 前端点击 `浏览 / 收藏 / 加购 / 购买`
2. 调用 `POST /api/recommendations/actions`
3. 后端成功写入日志
4. 前端把这次操作插入 `sessionActions`
5. 前端调用 `loadCustomerHome()`
6. 推荐、趋势、画像重新拉取
7. 页面更新：
   - `本次会话刚刚操作` 立刻显示新行为
   - `最近 5 条个人行为` 显示最新历史行为
   - 冷启动状态可能消失
   - 推荐与画像可能变化

## 7. 测试要求

### 7.1 前端测试

需要新增或更新测试，验证：

- 用户页出现 `本次会话刚刚操作`
- 用户页出现 `最近 5 条个人行为`
- 点击行为按钮后，会话区出现刚刚记录的行为
- 点击行为按钮后，历史最近行为区出现刷新后的行为记录
- 冷启动用户点击后，不再显示冷启动状态

### 7.2 后端测试

本次原则上不新增新的后端能力，因此不强制新增后端测试。

已有的用户行为写入测试和画像刷新测试已经覆盖主要链路：

- 用户行为能够写入
- 写入后画像会变化

除非实现过程中额外修改后端返回结构，否则本次后端无需扩展测试范围。

## 8. 价值说明

本次增强完成后，用户端将从“你只能看到结果变化”升级为“你既能看到刚刚做了什么，也能看到系统最近依据了什么”。

这会显著提升以下几方面：

- `演示清晰度`：老师能看见你刚刚操作的具体内容
- `系统可信度`：老师能看到画像依据不是凭空生成
- `链路完整性`：用户行为、画像分析、推荐结果之间的关系更完整
- `答辩表达`：更容易解释“系统如何从行为日志推导出画像与推荐”

## 9. 实现边界

本次只做最小且高价值的行为展示增强：

- 本地会话行为显示
- 历史最近行为显示
- 页面内即时刷新后的可视化反馈

本次明确不做：

- 行为数据持久化会话追踪
- 独立行为中心
- 复杂筛选器
- 后端新增历史行为查询 API

目标是优先服务答辩展示，而不是把用户端扩展成完整行为分析后台。
