# 商家端用户行为变化联动卡片设计

## 1. 目标

本次增强用于把用户端新补上的真实行为闭环，进一步联动到商家端首页，让商家能够直接看到“最近用户行为变化正在带来什么经营信号”。

当前系统已经具备：

- 用户端行为按钮：浏览、收藏、加购、购买
- 用户端最近行为展示
- 商家端经营分析与运营动作闭环

但现在两端还是相对独立的：

- 用户端能看到自己的画像和推荐变化
- 商家端能看到商品热度、冷门商品、风险和运营动作

还缺少一个明确的“桥梁模块”，告诉老师：

- 用户最近偏好在变什么
- 商家最近该关注哪些高意向商品

因此本次目标是在商家首页新增一个 `用户行为变化` 卡片，用最小改动把用户侧变化映射到商家侧经营视角。

## 2. 设计范围

本次只做商家首页内的一个新增分析卡片，不新增独立页面。

卡片包含两个子区域：

1. `最近用户偏好变化`
2. `最近高意向商品变化`

本次不做：

- 实时推送
- 独立用户行为分析页
- 多维筛选器
- 复杂时间范围选择器
- 用户明细列表
- 个体级用户隐私展示

目标是优先增强答辩时的“两端联动感”，而不是把商家端扩成完整用户行为后台。

## 3. 核心设计

### 3.1 卡片位置

放置在：

- `frontend/src/views/merchant/MerchantDashboard.vue`

作为商家首页中的一个独立 `dashboard-panel`，与 `热销机会`、`滞销风险`、`异常预警`、`AI 经营分析` 并列。

推荐位置：

- 放在商家分析区中部靠前，便于答辩时在“用户端操作完后切到商家端”第一时间看到变化。

### 3.2 模块结构

卡片标题：

- `用户行为变化`

卡片分为两个子区：

#### 子区一：最近用户偏好变化

从当前日志中提炼最近更活跃的用户偏好信号，重点看：

- 哪些类目近期更活跃
- 哪些行为类型更集中

展示示例：

- `运动鞋 · 收藏/加购行为上升`
- `智能设备 · 最近开始出现购买行为`

该子区重点回答：

- 最近用户更关注什么类型的商品
- 最近用户行为正在从“看”走向“想买”还是“已经买”

#### 子区二：最近高意向商品变化

从当前日志中提炼最近高意向商品信号，重点关注：

- 被收藏较多的商品
- 被加购较多的商品
- 被购买较多的商品

展示示例：

- `轻量跑鞋 · 最近加购 3 次`
- `透气训练T恤 · 最近收藏 2 次`
- `智能运动手环 · 最近购买 1 次`

该子区重点回答：

- 商家最近应该优先关注哪些具体商品
- 哪些商品正在形成较强成交意向

## 4. 后端设计

### 4.1 接口设计

新增一个商家侧接口：

- `GET /api/analytics/merchant/user-behavior`

该接口只允许商家访问。

返回结构分两部分：

```json
{
  "preference_changes": [
    {
      "category": "运动鞋",
      "top_action": "favorite",
      "action_count": 3,
      "summary": "运动鞋最近收藏行为更活跃"
    }
  ],
  "intent_products": [
    {
      "product_id": 1,
      "product_name": "轻量跑鞋",
      "top_action": "cart",
      "action_count": 2,
      "summary": "轻量跑鞋最近加购较多，值得重点关注"
    }
  ]
}
```

### 4.2 数据来源

继续复用现有日志来源：

- `simulation_memory_store.logs`

不新增新表，不改现有日志结构。

这样做的优势是：

- 用户端刚刚产生的行为会立即进入商家侧分析
- 不需要单独再维护一套“商家联动数据”
- 保持系统逻辑统一

### 4.3 聚合逻辑

#### 用户偏好变化聚合

按日志中的以下字段聚合：

- `category`
- `action_type`

只关注更能体现意向的行为：

- `favorite`
- `cart`
- `purchase`

输出规则：

- 每个类目选取当前最活跃的高意向行为
- 生成一条简短 summary
- 按意向强度和次数排序

#### 高意向商品变化聚合

按以下字段聚合：

- `product_id`
- `product_name`
- `action_type`

同样优先关注：

- `favorite`
- `cart`
- `purchase`

输出规则：

- 每个商品选取当前最强的高意向行为
- 返回最近更值得关注的商品列表
- 按行为权重和次数排序

### 4.4 权限边界

该接口只允许 `merchant` 调用：

- `merchant` 可以访问
- `customer` / `admin` 返回 `403`

这样保证接口语义清晰，避免把商家分析数据误暴露给其他角色。

## 5. 前端设计

### 5.1 商家首页展示方式

新增一个 `dashboard-panel`：

- 标题：`用户行为变化`
- 副标题：说明“该模块用于把最近用户行为信号转成商家可理解的经营变化”

内部用两列或上下两个列表呈现：

1. `最近用户偏好变化`
2. `最近高意向商品变化`

### 5.2 展示内容

每一项显示：

- 主标题
- 行为类型
- 简短总结

例如：

- `运动鞋`
  - `收藏 3 次`
  - `运动鞋最近收藏行为更活跃`

或：

- `轻量跑鞋`
  - `加购 2 次`
  - `轻量跑鞋最近加购较多，值得重点关注`

### 5.3 空状态

当目前日志中还没有足够高意向行为时，显示：

- `当前暂无明显用户偏好变化`
- `当前暂无明显高意向商品变化`

这样在冷启动或行为量较少时也不会显得页面出错。

### 5.4 联动方式

商家页不做自动推送，继续沿用现有手动刷新方式：

- 用户端完成若干操作
- 切换到商家端
- 点击 `刷新数据`
- 商家端重新拉取这个联动分析接口

这已经足够满足答辩展示需求，也避免引入不必要的复杂前后端同步机制。

## 6. 数据流

完整链路如下：

1. 用户端点击浏览、收藏、加购、购买
2. 后端写入日志
3. 用户端推荐和画像变化
4. 商家端刷新首页
5. 商家端请求 `GET /api/analytics/merchant/user-behavior`
6. 后端根据同一批日志聚合出偏好变化和高意向商品变化
7. 商家端卡片渲染新的用户行为变化结果

这条链路能直接体现：

- 用户侧行为不是孤立的
- 同一批行为日志也会反向影响商家经营判断

## 7. 测试要求

### 7.1 后端测试

需要新增或更新测试，验证：

- 商家可访问 `GET /api/analytics/merchant/user-behavior`
- 非商家访问返回 `403`
- 返回结果包含：
  - `preference_changes`
  - `intent_products`
- 当存在收藏/加购/购买行为时，接口能返回非空结果

### 7.2 前端测试

需要新增或更新测试，验证：

- 商家页出现 `用户行为变化`
- 商家页出现：
  - `最近用户偏好变化`
  - `最近高意向商品变化`
- mock 数据时能正确渲染类目变化和商品变化
- 空数据时能显示空状态文案

## 8. 价值说明

本次增强完成后，系统将首次在页面层面形成真正的“两端联动证据链”：

- 用户做出真实行为
- 用户端推荐与画像变化
- 商家端同步看到用户偏好变化和高意向商品变化

这对答辩的价值非常高，因为它直接回应了老师最关心的几个点：

- 不是单纯看板
- 有交互
- 有分析结果
- 有角色联动
- 有业务意义

## 9. 实现边界

本次只做最小商家联动增强：

- 新增一个商家端联动卡片
- 新增一个轻量分析接口
- 复用现有日志链路

本次明确不做：

- 实时推送
- 多商家复杂隔离
- 复杂行为分析后台
- 用户级细粒度追踪系统

目标是优先让系统“更像一个完整的电商智能分析系统”，而不是继续扩展成过重的平台。
