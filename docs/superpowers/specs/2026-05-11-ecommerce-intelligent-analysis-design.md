# 基于用户行为日志的电商智能分析与决策支持系统设计文档

## 1. 项目背景

当前已有的“电商日志分析”方案主要停留在“导入 CSV 后进行可视化展示”的层面，虽然具备基础的数据展示能力，但缺少持续性日志采集、数据分析建模、业务决策输出和多角色交互等核心环节，难以体现数据科学与大数据技术专业的系统设计与分析能力。

为解决上述问题，本项目重新设计为一个面向商家端、用户端和管理员端的前后端分离系统。系统以电商用户行为日志为核心数据对象，围绕日志生成、清洗、聚合、分析、预测、推荐和经营建议生成构建完整闭环，最终实现一个兼具工程实现、数据分析和业务决策能力的毕业设计系统。

## 2. 项目目标

本项目的总体目标是设计并实现一个“基于用户行为日志的电商智能分析与决策支持系统”，实现以下能力：

1. 构建持续生成的电商行为日志体系，而非依赖单次导入的静态数据文件。
2. 支持商家端、用户端、管理员端三类角色的差异化使用场景。
3. 对用户行为日志进行清洗、存储、特征构建和统计分析。
4. 实现商品热度评分、趋势预测、用户画像、推荐分析和异常检测。
5. 基于结构化分析结果生成面向商家和用户的决策建议。
6. 提供清晰的前后端分离架构和可演示的系统工作流，满足毕业设计答辩要求。

## 3. 系统定位

### 3.1 项目名称

推荐项目名称为：

- 基于用户行为日志的电商智能分析与决策支持系统

也可使用下列备用名称：

- 面向商家与用户双端的电商行为日志分析与热度预测系统
- 电商用户行为日志智能分析与可视化决策系统

### 3.2 系统定位说明

本系统不是单纯的数据报表展示平台，而是一个围绕“电商日志分析结果服务业务决策”的智能分析系统。

系统的核心定位为：

- 对管理员而言，系统用于维护基础数据、控制模拟任务和分析任务。
- 对商家而言，系统用于发现热销商品、低转化商品、地区偏好和经营风险，并给出补货、促销、优化详情页等建议。
- 对普通用户而言，系统用于查看热门趋势、获取个性化推荐、理解推荐理由，并辅助购物决策。

## 4. 系统角色与业务场景

### 4.1 商家端

商家端主要面向商品经营分析与决策支持，核心需求包括：

- 查看商品热度排行、潜力商品排行、滞销商品排行。
- 查看商品漏斗转化情况，例如浏览、点击、收藏、加购、购买之间的转化路径。
- 分析不同地区、不同类目、不同时间段的消费行为。
- 查看库存预警、经营异常和策略建议。
- 对重点商品执行促销、重点观察、优先补货等操作标记。

### 4.2 用户端

用户端主要面向趋势感知与推荐服务，核心需求包括：

- 浏览当前热销商品、趋势上升商品和推荐商品。
- 查看自身偏好画像，包括偏好类目、价格区间和活跃时段。
- 查看浏览、收藏、加购、购买等行为历史。
- 获取推荐理由说明，如“与你近期关注的运动鞋类商品相关”。

### 4.3 管理员端

管理员端主要负责系统运行和数据任务管理，核心需求包括：

- 管理用户、商家和商品基础信息。
- 管理日志模拟任务和分析任务。
- 查看系统日志、任务执行记录和运行状态。
- 初始化基础数据并维护系统运行环境。

## 5. 总体技术方案

### 5.1 技术路线

本系统采用“前后端分离 + 日志分析引擎 + 结构化建议输出”的技术路线。

- 前端：Vue 3 + Vite + Pinia + Vue Router + Element Plus + ECharts
- 后端：Flask + JWT + REST API
- 业务数据库：MySQL
- 日志存储与查询：Elasticsearch
- 分析计算：Python 统计分析与轻量机器学习
- 定时任务：APScheduler 或等效任务调度组件
- 大模型增强：基于结构化分析结果生成自然语言报告与建议

### 5.2 选择原因

采用该技术路线的原因如下：

1. Vue 3 和 Flask 组合轻量稳定，适合在有限时间内快速实现前后端分离系统。
2. MySQL 适合存储用户、商品、策略记录、任务记录等业务数据。
3. Elasticsearch 适合存储原始电商行为日志，支持条件检索和聚合分析。
4. Python 便于实现统计分析、聚类、时间序列预测和推荐逻辑。
5. 大模型可以作为上层解释层，将结构化分析结果转化为自然语言建议，提高系统展示效果与论文创新度。

## 6. 系统总体架构

系统采用分层架构，自上而下可分为五层：

### 6.1 表现层

表现层为前端三端界面，包括商家端、用户端和管理员端。前端负责展示分析结果、提供交互入口和触发业务操作。

### 6.2 接口层

接口层由 Flask 提供 RESTful API，负责用户认证、业务查询、任务控制和分析结果下发。

### 6.3 业务服务层

业务服务层负责用户管理、商品管理、日志模拟、数据清洗、分析计算、预测、推荐、建议生成等核心业务逻辑。

### 6.4 数据层

数据层由 MySQL 和 Elasticsearch 组成：

- MySQL 存储用户、商品、任务记录、建议记录、推荐记录等结构化业务数据。
- Elasticsearch 存储原始行为日志和部分聚合快照数据。

### 6.5 智能分析层

智能分析层由统计分析模块、机器学习模块、异常检测模块和大模型报告模块构成，为系统提供决策支持能力。

## 7. 数据来源与日志处理方案

### 7.1 数据来源

本系统不再将上传 CSV 作为唯一核心数据来源，而是采用“模拟日志为主，导入能力为辅”的方案。

数据来源包括：

1. 初始化生成的商品基础数据。
2. 初始化生成的用户基础数据。
3. 系统内置的电商日志模拟器持续生成行为日志。
4. 可选的商品主数据导入功能，用于补充商品信息。

### 7.2 日志字段设计

每条行为日志建议包含如下字段：

- `log_id`：日志唯一标识
- `user_id`：用户编号
- `merchant_id`：商品所属商家编号
- `product_id`：商品编号
- `product_name`：商品名称
- `category`：商品类目
- `brand`：品牌
- `price`：价格
- `action_type`：行为类型
- `region`：用户地区
- `device_type`：设备类型
- `source_channel`：来源渠道
- `session_id`：会话编号
- `stay_duration`：停留时长
- `is_new_user`：是否新用户
- `timestamp`：行为时间

其中 `action_type` 固定为如下取值之一：

- `view`
- `click`
- `favorite`
- `cart`
- `purchase`

### 7.3 日志处理流程

日志处理链路设计为：

`日志生成/采集 -> 数据清洗 -> 特征构建 -> 聚合统计 -> 预测分析 -> 建议生成 -> 前端展示`

具体步骤如下：

1. 日志模拟器按照用户、商品、类目、地区、时间段等规则生成行为数据。
2. 后端对原始日志进行清洗，包括空值处理、重复值去除、字段标准化和异常值修正。
3. 清洗后的原始日志写入 Elasticsearch。
4. 系统按时间窗口进行聚合，生成日级或小时级统计指标。
5. 分析引擎对聚合结果进行热度计算、趋势预测、用户分群、推荐计算和异常检测。
6. 最终结果以图表、表格、标签、文字建议等方式展示在三端页面中。

## 9. 关键分析指标与统计方法

### 9.1 描述统计

系统需要输出以下基础统计指标：

- PV
- UV
- 购买次数
- 客单价
- 收藏率
- 加购率
- 购买率

### 9.2 漏斗分析

构建如下转化路径：

`浏览 -> 点击 -> 收藏 -> 加购 -> 购买`

通过漏斗分析识别高流量低转化商品和潜在优化节点。

### 9.3 时间窗口统计

按小时、按天、按类目、按地区进行聚合，分析行为变化趋势和周期特征。

### 9.4 RFM 用户价值分析

以最近购买时间、购买频次、消费金额构建用户价值分层，用于识别高价值用户、低活跃用户和潜在流失用户。

### 9.5 同比与环比分析

对商品热度、购买量和转化率进行周期对比，用于识别上升和下降趋势。

## 10. 核心算法设计

### 10.1 商品热度评分

采用加权评分法构建商品热度分：

`hot_score = 1*view + 2*click + 3*favorite + 5*cart + 8*purchase`

设计理由：

1. 不同行为对购买意愿的代表性不同。
2. 购买行为权重最高，浏览行为权重最低。
3. 该方法可解释性强，适合作为论文中的核心基础模型。

该模块输出：

- 热销榜
- 潜力榜
- 滞销榜
- 上升榜

### 10.2 时间序列趋势预测

预测模块以商品日级热度分为输入，采用如下方法：

1. 移动平均，用于平滑短期波动。
2. 线性回归，用于预测未来 1 到 3 天热度趋势。
3. 如时间允许，可扩展 ARIMA 模型作为增强实验。

该模块输出：

- 趋势上升商品
- 趋势平稳商品
- 趋势下滑商品

### 10.3 用户画像与聚类分析

用户画像模块采用两类方法：

1. RFM 分析，用于进行用户价值分层。
2. KMeans 聚类，用于根据活跃度、消费金额、行为分布等特征划分用户群。

可得到的典型群体包括：

- 高活跃高消费用户
- 高频浏览低转化用户
- 偶发购买用户
- 低活跃沉默用户

### 10.4 推荐算法

推荐模块采用两层结构：

1. 规则推荐：基于用户偏好类目、热门商品和趋势商品进行推荐。
2. 简单协同过滤：基于相似用户或相似商品行为进行补充推荐。

推荐输出需附带推荐理由，增强结果可解释性。

### 10.5 异常检测

异常检测采用轻量可解释方案：

1. 阈值规则检测。
2. 环比突增检测。
3. Z-score 标准分数检测。

典型异常包括：

- 浏览量快速增长但购买转化较低。
- 收藏量较高但加购量较低。
- 某地区流量增长异常。

## 11. 大模型增强设计

### 11.1 接入原则

大模型不是底层核心分析模块，而是上层解释与表达模块。核心统计、预测、推荐和异常检测结果必须由本系统自身完成。

### 11.2 输入内容

传递给大模型的内容为结构化摘要，而非原始日志。例如：

- 商品热度分
- 连续多日趋势变化
- 各地区购买表现
- 当前主要问题
- 用户画像标签
- 推荐列表及依据

### 11.3 输出内容

大模型用于生成以下结果：

- 商家经营建议
- 商品分析摘要
- 运营周报
- 用户推荐理由
- 平台运行日报

### 11.4 设计价值

该设计能够突出“系统自主分析 + 大模型智能解释”的分工关系，避免答辩时被质疑“完全依赖大模型”。

## 12. 前端页面设计

### 12.1 商家端页面

商家端主要页面包括：

1. 商家首页：展示核心经营指标、热销商品、预警信息和趋势概览。
2. 商品热度中心：展示热销榜、潜力榜、滞销榜和商品搜索筛选。
3. 转化分析页：展示漏斗图和转化率对比。
4. 地区分析页：展示地区购买分布和地区类目偏好。
5. 策略建议中心：展示补货建议、促销建议、异常提醒和 AI 周报。
6. 商品管理页：支持商品维护、库存查看、促销状态管理。

### 12.2 用户端页面

用户端主要页面包括：

1. 用户首页：展示热销商品、猜你喜欢和上升趋势商品。
2. 推荐中心：展示推荐商品列表和推荐理由。
3. 趋势榜单页：展示热销榜、上升榜、收藏榜等。
4. 个人画像页：展示偏好类目、价格偏好、活跃时段和消费等级。
5. 行为历史页：展示浏览、收藏、加购和购买记录。

### 12.3 管理员端页面

管理员端主要页面包括：

1. 系统总览页：展示日志量、用户数、商品数、任务状态和系统运行情况。
2. 用户管理页：管理账号、角色、状态。
3. 商品管理页：初始化和维护商品数据。
4. 模拟任务管理页：启动、停止和配置日志模拟任务。
5. 分析任务管理页：执行聚合、预测、推荐和报告任务。
6. 系统日志页：查看错误日志、操作日志和任务执行日志。

## 13. 后端模块划分

后端按领域划分为以下模块：

- `auth`：登录、注册、权限控制、JWT 认证。
- `users`：用户资料、画像查询、行为摘要。
- `products`：商品信息、库存信息、促销状态维护。
- `simulation`：用户行为日志模拟任务管理。
- `etl`：数据清洗、字段标准化和聚合预处理。
- `analytics`：总览指标、漏斗、地区分析、类目分析、RFM 分析。
- `prediction`：热度趋势预测、上升榜和下滑榜生成。
- `recommendation`：推荐结果生成和推荐理由输出。
- `strategy`：商家经营建议和异常预警。
- `llm`：大模型报告生成和文本摘要。
- `admin`：任务调度、运行状态和系统资源管理。

## 15. 系统数据流设计

系统的数据流设计如下：

1. 管理员初始化商品、用户、类目和地区等基础数据。
2. 日志模拟器按设定规则生成行为日志。
3. 原始日志进入清洗流程后写入 Elasticsearch。
4. 聚合服务按时间窗口计算商品指标和用户指标。
5. 分析服务执行热度评分、漏斗分析、RFM、聚类、预测和异常检测。
6. 推荐服务和策略服务根据分析结果生成推荐列表与经营建议。
7. 大模型模块读取结构化摘要，生成自然语言报告。
8. 商家端、用户端和管理员端分别读取对应结果并展示。

1. 具备清晰的前后端分离结构。
2. 支持三角色权限隔离。
3. 模拟数据和分析结果具备可重复演示能力。
4. 核心分析逻辑具备可解释性。
5. 页面具备基本可用性和可视化表现力。
6. 系统支持答辩场景下的稳定运行和关键功能展示。

## 17. 最小可交付版本范围

为保证实现进度，最小可交付版本必须覆盖以下能力：

1. 三角色登录与权限控制。
2. 商品、用户、日志数据结构初始化。
3. 日志模拟器。
4. 行为统计总览。
5. 商品热度评分。
6. 漏斗分析。
7. 地区分析。
8. 热度趋势预测。
9. 商家建议输出。
10. 用户推荐与推荐理由。
11. 管理员任务管理页面。

以下能力可作为增强项，不纳入最小交付要求：

- 更复杂的协同过滤推荐算法。
- ARIMA 等更复杂的预测实验。
- 更丰富的 AI 周报与报告生成。
- 实时流式日志看板。

## 19. 风险与约束

项目实现中需注意以下风险：

1. 若过度追求复杂模型，可能导致实现进度和演示稳定性受影响。
2. 若过度依赖大模型，容易被质疑缺少自主分析能力。
3. 若页面过多但分析结果不够扎实，会再次落回“功能多但价值弱”的问题。

因此本项目的实现原则为：

- 先保证日志闭环、分析闭环和建议闭环成立。
- 先保证核心算法可解释、可实现、可展示。
- 大模型只作为增强层，不替代底层分析。

## 20. 结论

本设计文档明确了新毕业设计系统的定位、架构、数据流、算法方法、页面模块和最小交付范围。系统重点不再是简单的 CSV 可视化，而是围绕电商行为日志构建“数据生成、行为分析、趋势预测、推荐服务和决策支持”的完整闭环，能够更好体现数据科学与大数据技术专业的毕业设计要求。

## 21. 当前实现对应的运行说明

结合当前仓库中的最小可交付实现，运行方式说明如下：

1. 后端使用 Flask 启动，本地默认地址为 `http://127.0.0.1:5000`。
2. 前端使用 Vue 3 + Vite 启动，通过 `VITE_API_BASE_URL` 指向后端接口地址。
3. 当前演示版默认使用 SQLite 本地数据库完成数据持久化，便于答辩环境快速部署。
4. 系统启动后会自动写入演示账号、商品和任务数据，无需手工导入基础数据。
5. Elasticsearch 地址仍保留为扩展配置项，但最小演示闭环不依赖其实际部署。

建议的环境准备如下：

- Python `3.9+`
- Node.js `18+`
- npm `9+`

后端关键环境变量如下：

- `SECRET_KEY`
- `JWT_SECRET_KEY`
- `DATABASE_URL`
- `CORS_ORIGINS`
- `ES_URL`
- `AUTO_SEED_DEMO_DATA`

前端关键环境变量如下：

- `VITE_API_BASE_URL`
