# MySQL 日志持久化与分页预览设计

## 1. 目标

本次改动的目标是解决当前系统在日志存储与查看上的两个核心问题：

- 行为日志主要保存在内存中，服务重启后丢失，不能算真正持久化
- 管理员日志预览页只返回最近 20 条样例日志，无法查看批量生成的完整数据

本次改动完成后，系统应达到以下结果：

1. 基础业务数据与行为日志都能够落到 MySQL
2. 管理员日志预览页支持分页查看完整日志，而不是只看最近 20 条
3. 商家分析、用户画像、推荐与管理员算法链路都基于数据库中的真实日志数据

## 2. 范围

本次只做以下 2 类改动：

1. 行为日志持久化到 MySQL
2. 管理员日志预览接口改为分页查询

本次不做：

- 管理员页左右布局重排
- CSV 导出
- Elasticsearch、Kafka、Logstash 等额外中间件接入
- 日志检索高级筛选器
- 新增独立日志管理页面

## 3. 当前问题

### 3.1 日志未真正持久化

当前系统中：

- 用户、商品、任务等基础数据通过 SQLAlchemy 存在数据库
- 行为日志主要保存在 `simulation_memory_store.logs`

这种做法虽然适合快速演示，但存在明显缺点：

- 应用重启后日志丢失
- 无法直接以数据库查询方式查看完整日志
- 商家分析、推荐、画像和管理员链路都依赖运行期内存状态
- 论文和答辩中难以有说服力地说明“日志已经真正存储”

### 3.2 管理员页只显示 20 条样例

当前管理员日志预览接口只返回：

- 最近 20 条日志

这会导致：

- 即使快速生成 2000 条日志，页面上仍只能看到 20 条
- 用户无法确认系统是否真的生成了大量日志
- 页面更像“样例预览”，而不是“完整日志查看”

## 4. 总体方案

### 4.1 数据层方案

本次改动后，系统统一采用 SQLAlchemy 管理两类数据：

- 基础业务数据：用户、商品、任务
- 行为日志数据：新增行为日志表

运行环境使用 MySQL 作为主数据库。

测试环境仍可保留 SQLite 作为轻量回归环境，以避免测试依赖外部数据库服务。

### 4.2 过渡原则

本次不要求一次性删除所有内存日志逻辑，但要明确：

- MySQL 日志表是主数据源
- 管理员预览、商家分析、推荐、画像、算法链路后续都应优先基于数据库日志读取

如果保留内存列表，也只能作为兼容性过渡或运行期缓存，不能继续作为核心真实来源。

## 5. 数据模型设计

### 5.1 新增行为日志表

建议新增模型：

- `BehaviorLog`

建议字段至少包含：

- `id`
- `log_id`
- `user_id`
- `merchant_id`
- `product_id`
- `product_name`
- `category`
- `brand`
- `price`
- `action_type`
- `region`
- `device_type`
- `source_channel`
- `session_id`
- `stay_duration`
- `is_new_user`
- `timestamp`
- `created_at`

### 5.2 设计要求

字段设计应满足以下要求：

- 能完整覆盖当前内存日志结构
- 能支撑商家分析与用户侧推荐、画像需求
- 能支持管理员预览页按时间倒序分页查看

### 5.3 索引建议

建议至少考虑以下查询方向：

- `timestamp`
- `merchant_id`
- `user_id`
- `product_id`
- `action_type`
- `source_channel`

本次不要求做复杂联合索引优化，但至少要为常用查询路径保留扩展空间。

## 6. MySQL 接入策略

### 6.1 运行环境

后端配置中的默认数据库目前是 SQLite。

本次改动后，应支持通过：

- `DATABASE_URL`

显式指向 MySQL，例如：

- `mysql+pymysql://username:password@127.0.0.1:3306/ecommerce_analysis`

### 6.2 测试环境

为了不破坏现有 pytest 运行效率，测试环境仍可使用：

- `sqlite://`

也就是说：

- 生产 / 本地演示环境：MySQL
- 自动化测试环境：SQLite

这不影响论文和答辩中对系统数据库方案的描述，因为系统运行时已经支持并采用 MySQL。

### 6.3 依赖要求

如果当前环境缺少 MySQL 驱动，需要补充 Python 依赖，例如：

- `PyMySQL`

本次应把驱动依赖纳入后端运行要求。

## 7. 日志写入设计

### 7.1 模拟日志写入

管理员触发：

- 立即生成一批
- 快速生成 2000 条
- 定时模拟任务

时，生成出的日志不再只写入内存，而是批量写入 `BehaviorLog` 表。

### 7.2 用户真实行为写入

普通用户在推荐页执行：

- 浏览
- 收藏
- 加购
- 购买

后，系统应把该行为直接写入 `BehaviorLog` 表。

### 7.3 写入原则

写入设计应满足：

1. 模拟日志与真实前台行为使用统一日志表
2. 通过 `source_channel` 区分日志来源
3. 推荐与画像继续只认 `customer_page` 真实行为
4. 商家分析和管理员全局分析可读取完整日志

## 8. 日志读取设计

### 8.1 管理员预览接口改造

当前接口：

- `GET /api/admin/logs/preview`

本次保留路径，但改为支持分页参数，例如：

- `page`
- `page_size`

### 8.2 返回结构

建议返回结构至少包含：

- `summary.total_logs`
- `summary.latest_timestamp`
- `summary.action_type_count`
- `pagination.page`
- `pagination.page_size`
- `pagination.total_pages`
- `pagination.total_items`
- `logs`

### 8.3 默认行为

接口默认仍可使用较小的分页值，例如：

- 默认每页 20 条

但这 20 条只能是“第一页默认显示条数”，不能再等同于“系统只能查看 20 条”。

### 8.4 排序规则

建议默认按 `timestamp` 倒序返回，优先展示最新日志。

## 9. 关联模块改造方向

### 9.1 管理员算法链路

管理员算法链路中的：

- 日志输入
- 聚合摘要
- 加权评分
- 画像与推荐摘要
- 异常检测摘要
- 用户分层摘要

应逐步从 MySQL 日志表读取，而不是只依赖内存日志列表。

### 9.2 商家分析

商家页中的：

- 总览
- 事件级经营指标
- 热销 / 冷门商品
- 地区和类目热点
- 简化用户分层
- 异常预警

应基于数据库日志进行商家口径统计。

### 9.3 用户画像与推荐

用户侧：

- 画像
- 推荐
- 最近行为

继续遵循现有规则：

- 只读取 `source_channel=customer_page` 的真实前台行为

但数据来源应改为数据库，而不是只看内存列表。

## 10. 前端改动边界

本次前端只做必要改动：

1. 管理员日志预览页增加分页请求能力
2. 页面可显示当前页和总条数
3. 页面继续展示日志表格，但数据来自分页接口

本次不做：

- 页面重排
- 左右栏重构
- 日志多条件筛选
- 单独日志详情弹窗

## 11. 测试要求

### 11.1 后端测试

需要新增或更新测试，验证：

1. 模拟生成日志后，`BehaviorLog` 表中存在对应记录
2. 用户真实行为会写入 `BehaviorLog` 表
3. 管理员日志预览接口支持分页并返回正确总量
4. 推荐与画像在数据库日志来源下仍保持既有行为

### 11.2 前端测试

需要新增或更新测试，验证：

1. 管理员日志预览页能正确消费分页结构
2. 页面显示总日志数和当前页日志
3. 生成 2000 条后，页面不会再误导为“只有 20 条”

### 11.3 回归测试

修完后需要重点回归：

- 推荐行为记录
- 商家看板统计
- 管理员算法链路
- 日志预览页

避免出现“日志进库了，但分析结果仍然从旧内存读”的口径分裂问题。

## 12. 预期价值

本次改动完成后，系统在答辩中的解释链会明显更稳：

- 可以明确说明系统已经接入 MySQL
- 可以明确说明行为日志是持久化保存的
- 可以明确展示批量生成的日志并分页查看
- 推荐、画像、经营分析和算法链路都建立在真实数据库日志基础上

这会比“日志存在内存里、页面只预览 20 条”更符合老师对系统完整性和工程合理性的预期。
