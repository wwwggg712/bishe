# 用户端冷启动与回退推荐设计

## 1. 目标

本次增强用于修正当前用户端两个影响答辩说服力的问题：

- 用户没有任何现场操作时，页面却直接显示完整偏好画像，容易让老师认为系统结论缺乏依据。
- 个性化推荐结果在候选商品较少或已交互商品被过滤后容易为空，导致页面看起来像推荐无效。

本次设计的目标不是引入完整实时推荐系统，而是在当前架构基础上补齐最小但合理的闭环，让用户端能够区分“冷启动用户”和“已有行为用户”，同时避免推荐列表直接空白。

## 2. 设计范围

本次只做三项增强：

1. 用户画像增加冷启动状态判断
2. 推荐结果增加回退推荐模式
3. 用户页面增加数据来源与状态说明

本次不实现以下内容：

- 用户端浏览、点击、收藏、加购等实时行为采集
- 复杂推荐算法升级
- 推荐历史记录
- 用户画像动态实时刷新链路

这样可以用最小改动快速解决“看起来不合理”的问题，同时为后续补实时交互留出空间。

## 3. 核心设计

### 3.1 冷启动画像

后端画像接口继续使用当前登录用户在日志中的行为数据进行统计，但新增一个明确状态字段：

- `is_cold_start`

判定规则为：

- 当当前用户在日志中没有任何个人行为记录时，视为冷启动用户
- 当当前用户在日志中已有个人行为记录时，视为已有行为用户

冷启动用户返回的画像结果仍保留基础结构，但不再伪装成已经得到完整偏好结论。此时：

- `top_categories` 为空
- `top_actions` 为空
- `recent_activity` 为空
- `engagement_score` 为 `0`
- `top_active_hours` 为空
- `price_preference.price_band` 为 `暂无数据`
- `profile_tags` 为空
- `is_cold_start` 为 `true`

前端在 `is_cold_start=true` 时，不再把页面文案渲染成“你已经有完整画像”，而是明确展示：

- `冷启动用户`
- `暂无个人行为，当前仅展示基础账号信息`

### 3.2 回退推荐

当前推荐逻辑会过滤用户已交互商品。在商品候选较少时，这会让推荐结果为空。本次新增推荐模式字段：

- `mode`

取值为：

- `personalized`
- `fallback`

推荐逻辑调整如下：

1. 先按现有方式生成个性化推荐结果
2. 如果结果非空，返回 `mode=personalized`
3. 如果结果为空，则回退到“热门商品/趋势导向推荐”
4. 回退推荐不再要求用户已有个人偏好，只保留“当前热度较高、适合作为冷启动推荐”的说明

回退推荐目标不是假装懂用户，而是在没有足够个人行为数据时，给出可展示、可解释的默认推荐结果，避免用户页面出现空白。

### 3.3 页面状态说明

用户首页需要增加更明确的状态说明，避免老师把预置历史模拟数据和现场交互结果混为一谈。

页面显示分两种情况：

1. `已有行为用户`
   - 画像区显示正常偏好画像
   - 推荐区显示 `个性化推荐`
   - 可以展示已有偏好类目、活跃时段、消费倾向

2. `冷启动用户`
   - 画像区显示 `冷启动用户`
   - 推荐区显示 `冷启动推荐`
   - 文案说明当前推荐主要基于热门趋势和基础账号信息

页面需要增加一句简短说明，表明推荐和画像的数据来源。例如：

- `当前推荐基于你的历史行为日志生成`
- `当前处于冷启动阶段，推荐结果基于热门趋势商品生成`

## 4. 后端设计

### 4.1 画像接口

接口保持不变：

- `GET /api/users/portrait`

返回结构新增：

- `is_cold_start`

返回示意：

```json
{
  "user": {
    "id": 3,
    "nickname": "Customer Demo",
    "region": "华南"
  },
  "is_cold_start": true,
  "top_categories": [],
  "top_actions": [],
  "recent_activity": [],
  "engagement_score": 0,
  "top_active_hours": [],
  "price_preference": {
    "average_price": 0,
    "price_band": "暂无数据"
  },
  "profile_tags": []
}
```

### 4.2 推荐接口

接口保持不变：

- `GET /api/recommendations/me`

返回结构新增：

- `mode`

返回示意：

```json
{
  "mode": "fallback",
  "items": [
    {
      "product_id": 1,
      "product_name": "轻量跑鞋",
      "category": "运动鞋",
      "reason": "该商品近期热度较高，适合作为当前阶段的冷启动推荐"
    }
  ]
}
```

### 4.3 推荐服务逻辑

推荐服务继续保留当前“偏好类目 + 全局热度”的最小实现，但新增两个约束：

1. 如果用户没有行为日志，则直接进入回退推荐模式
2. 如果用户有行为日志，但候选商品全部被过滤导致结果为空，也进入回退推荐模式

回退推荐的排序基础可以继续复用现有全局热度分，不额外增加复杂算法。

## 5. 前端交互设计

### 5.1 用户首页推荐区

当前推荐区从“只有列表内容”改成“列表 + 状态说明”：

- 当 `mode=personalized` 时：
  - 标题保持 `猜你喜欢`
  - 增加说明：`基于你的历史行为偏好生成`

- 当 `mode=fallback` 时：
  - 标题显示 `冷启动推荐`
  - 增加说明：`当前暂无足够个人行为，先为你展示热门趋势商品`

### 5.2 用户首页画像区

当前画像区在冷启动状态下不再展示强结论，而是突出状态：

- 显示 `冷启动用户`
- 显示 `暂无个人行为，当前仅展示基础账号信息`
- 保留地区等基础信息
- 不再把空数据包装成真实偏好画像

### 5.3 AI 推荐解释

`AI 推荐解释` 按钮继续保留，但解释文案需要适配状态：

- 当 `mode=personalized` 时，解释文案可继续使用“你最近更偏好”
- 当 `mode=fallback` 时，解释文案应偏向“当前根据热门趋势与基础信息为你推荐”

本次不新增新接口，只允许前端根据当前推荐模式组织更合理的输入文案。

## 6. 测试要求

### 6.1 后端测试

需要新增或更新测试，验证：

- 当用户没有个人行为日志时，`/api/users/portrait` 返回 `is_cold_start=true`
- 当用户没有个人行为日志时，`/api/recommendations/me` 返回 `mode=fallback`
- 当用户已有行为日志且存在可推荐商品时，`/api/recommendations/me` 返回 `mode=personalized`

### 6.2 前端测试

需要新增或更新测试，验证：

- 冷启动状态下页面出现 `冷启动用户`
- 回退推荐状态下页面出现 `冷启动推荐`
- 页面不再显示“暂无推荐结果，可先生成更多行为数据后再查看”
- `AI 推荐解释` 在回退推荐场景下仍可正常显示结果

## 7. 价值说明

本次增强完成后，用户端将从“页面有数据但解释不清楚”升级为“明确区分冷启动与个性化阶段”。这样可以直接提升以下几点：

- 推荐结果不再因为候选集过小而看起来失效
- 用户画像不会在没有依据时显得突兀
- 答辩时可以清楚说明哪些结果来自个人行为，哪些结果来自冷启动回退逻辑
- 为后续接入用户实时交互留下自然演进路径

## 8. 实现边界

本次增强的目标是“让当前系统讲得通”，而不是“把推荐系统一次做完”。

因此本次实现边界明确为：

- 修正冷启动呈现
- 修正推荐空白问题
- 强化页面解释性

后续如果继续扩展，再进入下一阶段：

- 用户端行为采集
- 实时刷新画像
- 基于实时行为的推荐重排
