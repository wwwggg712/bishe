# 系统可信度修正设计

## 1. 目标

本次增强不是继续加新功能，而是修正当前系统中几处“能跑但解释不够严谨”的地方，降低答辩时被老师追问后难以自圆其说的风险。

本次只修正前 4 项高优先级问题：

1. 商家账号混入用户行为日志
2. 商家端“最近用户行为变化”实际上不是最近
3. 商家端联动摘要出现英文动作词
4. 用户画像中的“消费倾向”口径不合理

本次明确不处理：

- 冷启动阈值升级

这样可以优先解决最影响系统可信度的问题，同时避免本轮改动范围过大。

## 2. 问题定义

### 2.1 商家账号混入用户行为日志

当前模拟日志生成时，会把 `customer` 和 `merchant` 都作为行为主体随机写入日志。这会导致：

- 用户推荐、画像、RFM、UV 等统计结果可能被商家账号污染
- 电商“用户行为分析”里混入非消费者角色

这在业务语义上是不合理的。

### 2.2 “最近变化”并不是真的最近

当前商家端 `用户行为变化` 卡片虽然文案写的是“最近用户偏好变化”“最近高意向商品变化”，但后端聚合实际上直接扫全量历史日志，没有时间窗口过滤。

这会导致：

- 新近行为不容易在商家端明显体现
- “最近”一词与实现不一致

### 2.3 商家端联动摘要中英混杂

当前后端 summary 直接使用原始动作值：

- `favorite`
- `cart`
- `purchase`

因此会出现类似：

- `运动鞋最近cart行为更活跃`

页面文字不够自然，也不利于答辩展示。

### 2.4 消费倾向口径不合理

当前用户画像中的 `price_preference.average_price` 是按所有行为日志价格平均得到的，包括：

- 浏览
- 收藏
- 加购
- 购买

这更像“看过什么价位的商品”，而不是“真正的消费倾向”。

## 3. 设计范围

本次只做以下修正：

1. 模拟行为用户池仅保留 `customer`
2. 商家行为联动分析增加“最近窗口”
3. 商家联动摘要统一中文动作词
4. 用户消费倾向价格带改为高意向行为口径

本次不做：

- 冷启动切换阈值调整
- 推荐算法大改
- 管理员页增强
- 新页面、新图表、新筛选器

## 4. 具体设计

### 4.1 修正一：模拟日志只使用 customer

调整位置：

- `backend/app/services/simulation_service.py`

当前 `generate_once_from_db()` 中用户查询改为：

- 只查询 `customer`

即：

- `User.role == "customer"`

这样后续所有依赖模拟日志的分析能力都会自动受益，包括：

- 推荐
- 画像
- RFM
- 趋势
- 商家联动分析

### 4.2 修正二：商家联动分析增加最近窗口

调整位置：

- `backend/app/services/analytics_service.py`

为 `build_merchant_user_behavior()` 增加固定“最近时间窗口”过滤。

本次推荐采用：

- `最近 24 小时`

原因：

- 答辩时最好解释
- 不需要引入前端筛选参数
- 能明显体现“用户刚刚做的行为”会影响商家端结果

聚合时只统计：

- 属于当前商家
- 行为类型为高意向动作：`favorite/cart/purchase`
- 且 `timestamp >= latest_timestamp - 24h`

这里的 `latest_timestamp` 取当前日志中的最大时间，而不是系统当前时间。这样即使测试或演示日志是固定时间，也能正确计算“最近窗口”。

### 4.3 修正三：商家联动摘要统一中文动作词

调整位置：

- `backend/app/services/analytics_service.py`

新增动作标签映射：

- `favorite -> 收藏`
- `cart -> 加购`
- `purchase -> 购买`

summary 改为中文表达，例如：

- `运动鞋最近收藏行为更活跃`
- `轻量跑鞋最近加购较多，值得重点关注`

前端继续保留当前中文格式化逻辑，但后端 summary 自身也必须是中文，确保：

- 前后端口径一致
- 测试 mock 与真实返回一致
- 页面文案可直接展示

### 4.4 修正四：消费倾向改为高意向行为口径

调整位置：

- `backend/app/services/prediction_service.py`

当前平均价格基于全部用户日志计算。本次改为分层口径：

1. 如果用户存在 `purchase` 行为：
   - 只基于 `purchase` 的商品价格计算平均价格
2. 如果没有 `purchase`，但存在 `cart` 或 `favorite`：
   - 基于 `cart + favorite` 的商品价格计算平均价格
3. 如果连上述高意向行为都没有：
   - 返回 `0 / 暂无数据`

这样 `price_preference.price_band` 更能体现真实购买倾向，而不是被浏览行为大量稀释。

## 5. 前端影响

前端无需新增页面。

预计只需做轻量调整：

- 商家端 `用户行为变化` 卡片的描述文案可继续保留“最近”表述
- 如有必要，增加一句说明：`基于最近 24 小时高意向行为聚合`
- 现有前端动作中文格式化逻辑继续保留

主要收益来自后端数据本身更合理，而不是前端结构变化。

## 6. 测试要求

### 6.1 后端测试

需要新增或更新测试，验证：

1. 模拟日志生成用户只来自 `customer`
2. 商家联动接口只统计最近窗口中的行为
3. 商家联动 summary 使用中文动作词
4. 用户画像消费倾向优先基于 `purchase`
5. 没有 `purchase` 时退回 `cart + favorite`
6. 没有高意向行为时返回 `暂无数据`

### 6.2 前端测试

需要新增或更新测试，验证：

1. 商家联动卡片真实后端风格文案能正常显示中文 summary
2. 如补充了最近窗口说明，则同步断言对应文案

## 7. 价值说明

本次修正后，系统在答辩时会更容易讲得通：

- 用户行为日志只代表消费者，而不是混入商家账号
- 商家端“最近变化”真正反映最近行为
- 商家联动文案不再中英混杂
- 用户消费倾向更接近真实购买意图

这些修正不会明显增加页面复杂度，但会明显提升系统语义严谨性和答辩可信度。

## 8. 实现边界

本次只做高优先级可信度修正，不继续扩展功能面。

不在本次范围内的内容包括：

- 冷启动阈值升级
- 推荐算法升级
- 实时推送
- 更多新页面和新角色能力

目标是先把当前系统“讲得通、解释稳”，再考虑下一轮增强。
