# 商家端统计口径修正设计

## 1. 目标

本次修正的目标是解决商家首页当前一个被单商家演示掩盖的问题：

- 商家首页大部分统计接口仍然直接使用全局日志
- 只有“用户行为变化”接口做了 `merchant_id` 过滤
- 当前因为演示数据里主要只有一个商家，这个问题不明显
- 一旦出现多商家数据，商家首页就会看到并不完全属于自己的经营结果

本次改动后，商家首页中的核心经营统计都必须基于当前登录商家的日志生成。

## 2. 修正范围

本次只修商家首页使用到的统计口径，不改页面结构，不新增新页面。

纳入本次范围的接口包括：

- `GET /api/analytics/overview`
- `GET /api/analytics/funnel`
- `GET /api/analytics/regions`
- `GET /api/analytics/categories`
- `GET /api/analytics/products/hot`
- `GET /api/analytics/products/cold`
- `GET /api/analytics/users/rfm`

以下接口本次不需要重做，因为它们已经具备商家视角：

- `GET /api/analytics/merchant/user-behavior`

## 3. 目标语义

本次修正后，商家首页里的以下内容都应该回答同一个问题：

- “这是当前商家的经营情况”

而不是：

- “这是整个平台所有日志混在一起的情况”

也就是说：

- 总览指标是当前商家的总量
- 漏斗是当前商家的漏斗
- 类目热点是当前商家商品对应的类目热点
- 热销和冷门商品是当前商家自己的商品
- 用户价值分层是和当前商家发生过行为的用户

## 4. 设计原则

本次采用以下原则：

1. 接口路径不变
2. 前端调用方式尽量不变
3. 商家请求自动收敛到 `merchant_id` 视角
4. 管理员或非商家场景本次不扩展重构

这样做的原因是：

- 当前页面结构和 API 已经稳定
- 本次目标是修统计口径，而不是再扩接口体系
- 尽量让修正集中在后端服务层和测试层

## 5. 实现思路

### 5.1 服务层统一过滤

在 `AnalyticsService` 中增加统一的商家日志过滤能力，例如：

- 输入：`merchant_id`, `logs`
- 输出：仅保留 `log["merchant_id"] == merchant_id` 的日志

随后商家首页所依赖的统计构建函数都基于过滤后的日志计算。

### 5.2 路由层传递当前商家身份

`/api/analytics/*` 这些接口本次继续保留原路径，但在处理商家请求时，需要把当前 JWT 中的商家身份传给服务层。

也就是说，调用流程变成：

1. 读取当前登录角色与用户 ID
2. 如果当前是商家，则把 `merchant_id` 传给服务层
3. 服务层先过滤日志，再做统计

### 5.3 已有商家行为接口保持不变

`/api/analytics/merchant/user-behavior` 目前已经是按商家过滤的，本次不重复改造，只保持与其他接口的口径一致。

## 6. 各模块预期变化

### 6.1 总览指标

修正后：

- `pv`
- `uv`
- `purchase_count`
- `top_products`
- `regions`

都只能来自当前商家相关日志。

### 6.2 漏斗

修正后：

- `view / click / favorite / cart / purchase`

只统计当前商家的相关行为。

### 6.3 类目与商品

修正后：

- `categories`
- `hot_products`
- `cold_products`

只统计属于当前商家的商品行为。

### 6.4 用户价值分层

修正后：

- `build_user_rfm()`

只基于当前商家相关日志识别用户价值，不再把别的商家用户行为混进来。

## 7. 前端影响

本次不要求前端重排页面，也不要求新增参数。

预期前端层面的影响是：

- `frontend/src/api/analytics.js` 保持现有调用方式
- `MerchantDashboard.vue` 保持现有渲染结构
- 页面数字会自然变成当前商家自己的统计结果

也就是说，这次是“后端口径修正”，不是“前端功能扩展”。

## 8. 测试要求

### 8.1 后端测试

需要新增或更新测试，验证：

1. 混合两个商家的日志时，商家访问总览接口只能看到自己的指标
2. 热销与冷门商品只包含当前商家的商品
3. 类目热点只包含当前商家商品对应的类目
4. RFM 只统计当前商家相关用户行为
5. 既有 `merchant/user-behavior` 逻辑不受影响

### 8.2 前端测试

本次前端只需要保持商家页既有测试继续通过。

如果需要新增测试，重点验证：

- 商家页在 merchant-scoped mock 数据下仍能正常渲染

## 9. 范围边界

本次只做：

- 商家首页核心统计改为 `merchant_id` 口径

本次不做：

- 管理员端全局统计重构
- 新增 merchant 专用新接口路径
- 多商家切换 UI
- 商家策略接口重构
- 趋势与异常接口的作用域扩展

## 10. 预期价值

本次改动后，商家首页的经营含义会真正成立：

- 商家看到的是自己的经营数据
- 而不是平台混合数据在单商家场景下的“看起来没问题”

这样在答辩时也更能自洽：

- 用户端强调“只认真实操作”
- 商家端强调“只看当前商家”

前后台两端的数据语义都会更合理。
