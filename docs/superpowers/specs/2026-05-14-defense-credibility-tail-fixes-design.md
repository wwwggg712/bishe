# 答辩可信度收尾修正设计

## 1. 目标

本次修正的目标是收掉当前系统里最后几处容易在答辩时被追问穿的问题：

- 商家经营统计接口虽然已经支持 `merchant_id` 口径，但主接口仍未统一做商家角色限制
- 商家页异常预警仍然使用全局异常口径，与商家首页其它模块不一致
- 商家页“成交转化率”使用的分母不是浏览量，指标定义不自洽
- AI 经营分析与今日经营简报仍可能使用不一致的热度值和转化率口径

本次改动后，商家首页在权限、数据口径、指标定义和 AI 摘要输入上都应保持一致。

## 2. 修正范围

本次只修以下 4 项：

1. 商家 analytics 主接口统一加商家角色限制
2. 商家异常预警改为商家口径
3. 商家总览与转化率指标定义修正
4. AI 经营分析与今日经营简报输入口径统一

本次不做：

- 新增一套全新的 merchant dashboard API
- 管理员全局统计重构
- 用户端新增功能
- 异常算法重写
- 商家首页布局重排

## 3. 权限修正

### 3.1 当前问题

以下接口虽然已经支持商家作用域过滤，但当前只有登录校验，没有显式的商家角色限制：

- `/api/analytics/overview`
- `/api/analytics/funnel`
- `/api/analytics/regions`
- `/api/analytics/categories`
- `/api/analytics/products/hot`
- `/api/analytics/products/cold`
- `/api/analytics/users/rfm`

这会导致普通用户登录后，仍可能访问经营统计接口。

### 3.2 新规则

上述接口统一只允许商家访问。

如果当前 JWT 角色不是 `merchant`，则返回 `403`。

### 3.3 目标语义

修正后，商家经营数据的语义应明确为：

- 只有商家能看
- 而且看到的是自己的经营数据

## 4. 商家异常预警修正

### 4.1 当前问题

商家页当前加载异常预警时，仍然调用全局异常接口，因此：

- 商家页其它模块是商家口径
- 但异常预警可能是平台全局口径

这会让首页内部语义不一致。

### 4.2 新规则

商家页使用的异常预警必须切换为商家口径。

推荐实现方式为：

- 在异常服务中增加可选 `merchant_id` 过滤能力
- 商家请求异常接口时自动传入当前商家身份

这样可以保留现有接口结构，避免再扩一套重复接口。

### 4.3 结果

修正后，商家页里：

- 热销机会
- 滞销风险
- 类目热点
- 用户价值分层
- 异常预警

都属于当前商家同一口径。

## 5. 指标定义修正

### 5.1 当前问题

目前商家总览中的 `pv` 实际统计的是日志总数，也就是行为总量，而不是浏览量。

但前端又把：

- `purchase_count / pv`

解释成：

- 成交转化率
- 购买行为 / 浏览行为

这在定义上不成立。

### 5.2 新规则

商家总览需要区分至少两类指标：

- `behavior_count`：行为总量
- `view_count`：浏览量

同时：

- `purchase_count`：购买行为次数
- `purchase_rate = purchase_count / view_count`

### 5.3 前端展示要求

前端卡片文案也要同步调整：

- 行为总量 -> 对应 `behavior_count`
- 浏览量 -> 对应 `view_count`
- 成交笔数 -> 对应 `purchase_count`
- 成交转化率 -> 对应 `purchase_count / view_count`

如果 `view_count == 0`，则转化率显示 `0.0%`。

## 6. AI 与经营简报口径统一

### 6.1 当前问题

商家页目前在生成：

- AI 经营分析
- 今日经营简报

时，存在以下风险：

- 热度值可能不完全一致
- 转化率可能继续使用错误分母
- 异常数可能仍来自非统一口径

### 6.2 新规则

两类摘要都必须基于同一组 merchant-scoped 指标：

- 同一重点商品
- 同一热度值
- 同一浏览量分母下的转化率
- 同一商家口径的异常预警数
- 同一类目热点来源

### 6.3 结果

修正后，页面展示、卡片指标、AI 摘要、经营简报之间的解释应保持一致，不再出现“页面一个口径、AI 另一个口径”的情况。

## 7. 实现原则

本次实现遵循以下原则：

1. 尽量复用现有接口路径
2. 尽量不改动商家页整体结构
3. 把口径修正在后端服务层与路由层收口
4. 把前端修改控制在字段消费与文案同步
5. 用测试明确锁定权限、作用域和指标定义

## 8. 测试要求

### 8.1 后端测试

需要新增或更新测试，验证：

1. 非商家角色访问商家 analytics 主接口返回 `403`
2. 商家异常预警只返回当前商家相关异常
3. 总览中的行为总量、浏览量、成交笔数和转化率口径正确
4. 既有商家行为联动、热销/冷门、RFM 等功能不受影响

### 8.2 前端测试

需要新增或更新测试，验证：

1. 商家页总览卡片展示新的指标口径
2. 商家页异常预警继续正常显示，但数据来自商家口径
3. AI 经营分析与今日经营简报在同一套口径下仍可正常生成

### 8.3 最终复查

修完后需要再做一轮可信度复查，重点核对：

- 角色隔离是否成立
- 商家页内部口径是否统一
- 指标名称与实际含义是否一致
- AI 摘要是否与页面展示自洽

## 9. 预期价值

本次修正完成后，系统在答辩中的解释链会更完整：

- 用户端：只认真实前台操作
- 商家端：只看当前商家数据
- 商家异常：只看当前商家风险
- 商家指标：定义自洽
- AI 解释：和页面口径一致

这样老师即使从权限、异常来源、指标定义或 AI 可信度切入追问，也更难再找到明显漏洞。
