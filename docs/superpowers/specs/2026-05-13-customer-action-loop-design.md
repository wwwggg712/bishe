# 用户端真实交互驱动推荐与画像变化设计

## 1. 目标

本次增强的目标是让用户端从“只展示推荐结果和画像”升级为“用户可以真实操作，并立即影响推荐与画像结果”的状态。这样可以解决当前系统中用户端仍然偏静态的问题，让答辩时能够直接展示：

- 用户先是冷启动状态
- 用户对商品进行浏览、收藏、加购、购买
- 系统记录真实行为日志
- 推荐结果与用户画像随行为变化

本次重点不是实现完整电商交易流程，而是补齐“用户行为 -> 日志 -> 分析结果变化”的展示闭环。

## 2. 设计范围

本次只做以下四类用户行为：

1. `浏览`
2. `收藏`
3. `加购`
4. `购买`

这些行为统一作用于用户首页推荐商品卡片。

本次不实现以下内容：

- 真实购物车页面
- 真实订单页面
- 支付流程
- 库存扣减
- 商品详情页跳转
- 用户历史行为独立管理页

目标是用最小改动，把“用户真的进行了操作”这件事体现出来，并让画像和推荐发生看得见的变化。

## 3. 交互方案

### 3.1 页面位置

行为按钮放在用户首页 `frontend/src/views/customer/CustomerHome.vue` 的推荐商品卡片中。每个推荐商品下方都显示一组操作按钮：

- `浏览`
- `收藏`
- `加购`
- `购买`

这样用户在当前页面就可以完成交互，不需要跳转其他页面，演示时也更直接。

### 3.2 操作后的页面表现

用户点击任意按钮后，页面执行以下流程：

1. 调用后端行为记录接口
2. 写入一条该用户对应的行为日志
3. 前端显示简短成功反馈
4. 自动重新拉取：
   - 用户推荐
   - 趋势商品
   - 用户画像
5. 页面内容更新，体现行为后的变化

这种设计能让老师现场看到“我刚点了一次收藏，系统就把它记进去了，并改变了画像与推荐结果”，比纯静态看板更有说服力。

### 3.3 冷启动衔接

本次功能直接建立在前一轮“冷启动与回退推荐”之上。

行为发生前：

- 页面可能显示 `冷启动用户`
- 推荐区可能显示 `冷启动推荐`

行为发生后：

- 如果这是该用户第一条个人行为日志，则用户应从冷启动状态切换为非冷启动状态
- 推荐区可以从 `冷启动推荐` 转为 `猜你喜欢`
- 画像区可以开始展示偏好类目、活跃时段、参与度得分等真实结果

## 4. 后端设计

### 4.1 接口设计

新增一个最小行为记录接口：

- `POST /api/recommendations/actions`

该接口只允许普通用户调用。

请求体：

```json
{
  "product_id": 1,
  "action_type": "favorite"
}
```

其中 `action_type` 允许值为：

- `view`
- `favorite`
- `cart`
- `purchase`

返回结果：

```json
{
  "message": "用户行为记录成功",
  "log": {
    "user_id": 3,
    "product_id": 1,
    "product_name": "轻量跑鞋",
    "action_type": "favorite"
  }
}
```

### 4.2 存储方式

本次不额外增加数据库表，继续复用现有的内存日志存储：

- `simulation_memory_store.logs`

原因是当前系统本身的分析、推荐、画像已经都依赖这条链路，新增用户行为时只需要把新行为以同样结构追加进去，就能立即被已有分析逻辑消费。

### 4.3 服务逻辑

需要新增一个最小的用户行为记录方法，职责只有三件事：

1. 校验当前用户身份是否为 `customer`
2. 根据 `product_id` 找到商品
3. 生成与现有模拟日志格式一致的一条日志并写入 `simulation_memory_store.logs`

日志字段保持与现有模拟日志一致，包括：

- `log_id`
- `user_id`
- `merchant_id`
- `product_id`
- `product_name`
- `category`
- `brand`
- `price`
- `action_type`
- `region`
- `device_type`
- `source_channel`
- `session_id`
- `stay_duration`
- `is_new_user`
- `timestamp`

这样后续画像、推荐、趋势、AI 解释都不需要重新适配另一套日志结构。

### 4.4 权限边界

该接口只允许普通用户访问。

- `customer` 可以记录行为
- `merchant` / `admin` 调用时返回 `403`

这能保证语义清晰，避免商家或管理员通过错误入口写入用户行为日志。

## 5. 前端设计

### 5.1 推荐卡片行为按钮

当前推荐卡片下新增 4 个按钮：

- `浏览`
- `收藏`
- `加购`
- `购买`

按钮设计要求：

- 使用当前已有按钮风格，保持页面一致性
- 点击后显示短暂 loading 状态
- 防止重复点击造成体验混乱

### 5.2 点击后的刷新策略

每次操作成功后，不做本地硬编码改值，而是统一重新拉取已有接口结果：

- `fetchMyRecommendations()`
- `fetchTrendProducts()`
- `fetchPreferenceProfile()`

原因是这样最稳，不需要在前端手动维护推荐结果和画像推导逻辑，也更符合当前系统“后端负责分析、前端负责展示”的结构。

### 5.3 反馈方式

操作成功后页面需要有可见反馈，至少包含：

- 当前商品已记录某行为

例如：

- `已记录收藏行为`
- `已记录加购行为`

反馈可以以简短状态文案的形式放在推荐模块中，无需引入复杂通知系统。

### 5.4 与 AI 推荐解释的关系

`AI 推荐解释` 按钮继续保留，不新增入口。

本次增强后，用户先点击若干行为按钮，再点击 `AI 推荐解释`，就能让老师看到 AI 解释不再只是基于预置状态，而是能结合刚刚产生的真实用户行为。

本次不要求 AI 文案逐条点名刚才做了什么动作，但要求它至少能够因为推荐和画像变化而输出不同结果。

## 6. 数据流

用户点击推荐商品行为按钮后的完整链路如下：

1. 前端触发 `POST /api/recommendations/actions`
2. 后端写入一条用户行为日志到 `simulation_memory_store.logs`
3. 前端请求最新推荐结果
4. 前端请求最新趋势商品
5. 前端请求最新用户画像
6. 页面重新渲染
7. 用户看到冷启动状态消失、画像更新、推荐结果变化

这条链路的重点不是复杂算法，而是“当前页内的即时可见变化”。

## 7. 测试要求

### 7.1 后端测试

需要新增或更新测试，验证：

- 普通用户可以成功写入 `view/favorite/cart/purchase`
- 非普通用户调用时返回 `403`
- 写入行为后，用户画像不再是冷启动
- 写入行为后，推荐接口可以返回新的推荐模式或新推荐结果

### 7.2 前端测试

需要新增或更新测试，验证：

- 推荐卡片出现 4 个行为按钮
- 点击按钮后调用行为记录接口
- 点击按钮后重新加载推荐、趋势、画像
- 冷启动用户点击后页面不再显示 `冷启动用户`
- 页面可以显示行为记录成功反馈

## 8. 价值说明

本次增强完成后，用户端将从“静态展示推荐结果”升级为“可以现场操作并改变分析结果”的系统。其价值主要体现在：

- 更符合老师要求的“有行为、有交互、有结果变化”
- 推荐与画像的可信度更强，不再像纯静态预置数据
- 可以和商家端动作闭环形成对照，体现系统两端都具备行为驱动能力
- 为后续继续扩展商品详情页、历史行为记录、真实用户路径分析奠定基础

## 9. 实现边界

本次只做：

- 记录用户行为
- 当前页刷新推荐与画像
- 展示行为后的变化

本次明确不做：

- 完整电商前台流程
- 多页面联动
- 复杂状态管理重构
- 订单系统
- 支付系统

目标是优先满足答辩展示价值和系统可信度，而不是把用户端扩成完整商城。
