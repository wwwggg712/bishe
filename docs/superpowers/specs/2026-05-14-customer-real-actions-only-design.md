# 用户端仅基于真实操作设计

## 1. 目标

本次修正的目标是解决用户端当前最核心的可信度问题：

- 用户没有亲自操作，却已经出现较完整的推荐和画像
- 用户界面无法触发的行为类型，也会出现在“高频行为”中
- 用户页一边强调“真实操作闭环”，一边又混入系统批量生成的模拟历史

本次改动后，用户端与“我本人真实做过什么”相关的内容，只允许基于前台真实操作日志生成。

## 2. 设计原则

本次采用的原则是：

1. 用户推荐只认真实前台操作
2. 用户画像只认真实前台操作
3. 用户最近行为只认真实前台操作
4. 平台级趋势内容可以继续使用全局日志

这样划分的原因是：

- 推荐、画像、最近行为属于“个人语义”，必须对用户本人负责
- 趋势商品属于“平台语义”，可以继续展示整体热度变化

## 3. 日志来源定义

当前系统里已经存在两类来源：

### 3.1 真实前台操作

由用户在前台点击以下按钮产生：

- `view`
- `favorite`
- `cart`
- `purchase`

该类日志由 `record_customer_action()` 写入，并带有：

- `source_channel = "customer_page"`

### 3.2 系统模拟历史

由管理员或定时模拟任务批量生成，来源包括：

- `homepage`
- `search`
- `recommendation`
- `campaign`

这类日志允许继续存在于系统中，用于平台趋势、商家分析和整体演示，但不再参与“个人语义”计算。

## 4. 推荐逻辑调整

### 4.1 当前问题

当前 `GET /api/recommendations/me` 会读取该用户的全部日志，不区分来源。

这会导致：

- 用户没有在前台点击过，也会像已有大量个人行为
- 模拟历史会直接改变个性化推荐结果

### 4.2 新规则

`GET /api/recommendations/me` 只允许基于当前用户且 `source_channel == "customer_page"` 的日志做个性化推荐。

如果没有真实前台操作日志，则：

- 直接进入冷启动
- 返回 `mode = fallback`
- 展示冷启动推荐

### 4.3 结果

这样一来，用户推荐的含义会变得明确：

- 没点过，就冷启动
- 点过，才个性化

## 5. 画像逻辑调整

### 5.1 当前问题

当前 `GET /api/users/portrait` 基于该用户全部日志生成画像。

这会导致：

- `top_categories`
- `top_actions`
- `recent_activity`
- `price_preference`
- `top_active_hours`

都可能来自系统模拟日志，而不是用户本人真实前台操作。

### 5.2 新规则

`GET /api/users/portrait` 只允许基于当前用户且 `source_channel == "customer_page"` 的日志生成画像。

如果没有真实前台操作日志，则返回冷启动画像：

- `is_cold_start = true`
- `top_categories = []`
- `top_actions = []`
- `recent_activity = []`
- `engagement_score = 0`
- `top_active_hours = []`
- `price_preference.price_band = "暂无数据"`

### 5.3 页面含义

这样一来：

- “最近 5 条画像依据”就真的等于用户本人前台真实点过的记录
- “行为偏好”就真的来自用户本人前台真实行为
- 页面不再需要靠解释文案去掩盖数据来源混杂的问题

## 6. 趋势商品处理

本次不调整趋势商品接口。

原因是趋势商品的语义是：

- 平台近期热度上升的商品

它本身就不是“你个人的历史行为”，所以允许继续基于全局日志生成。

这也可以保留用户页在冷启动状态下的内容密度，避免页面在第一次进入时过空。

## 7. 前端表现要求

本次不重排用户页结构，只调整由后端语义变化带来的展示结果。

预期表现如下：

### 7.1 无真实操作时

- 推荐区显示冷启动推荐
- 画像区显示冷启动用户
- “本次会话刚刚操作”为空
- “最近 5 条画像依据”为空

### 7.2 有真实操作时

- 推荐区从冷启动逐步转向个性化
- 画像区显示真实偏好变化
- “本次会话刚刚操作”继续展示本轮会话即时反馈
- “最近 5 条画像依据”展示历史真实操作日志

## 8. 测试要求

### 8.1 后端测试

需要新增或更新测试，验证：

1. 当某用户只有模拟历史、没有 `customer_page` 日志时：
   - 推荐接口返回 `fallback`
   - 画像接口返回 `is_cold_start = true`
2. 当混合存在模拟历史和 `customer_page` 日志时：
   - 推荐只依据 `customer_page` 计算偏好
   - 画像只依据 `customer_page` 生成 `top_actions` 和 `recent_activity`
3. 趋势接口不受本次调整影响

### 8.2 前端测试

需要新增或更新测试，验证：

1. 只有模拟历史时，用户首页仍显示冷启动
2. 记录一次真实前台行为后，用户首页退出冷启动
3. 画像页中的行为偏好和最近行为只反映真实前台行为

## 9. 范围边界

本次只做：

- 用户推荐来源收敛到真实前台操作
- 用户画像来源收敛到真实前台操作
- 用户最近行为来源收敛到真实前台操作

本次不做：

- 商家侧接口隔离
- 模拟器生成策略调整
- 新增日志来源配置后台
- 趋势算法重构

## 10. 预期价值

本次改动后，用户端会从“看起来像合理”变成“数据来源本身就合理”。

答辩时可以更直接地说明：

- 用户页不是展示系统替你造出来的个人历史
- 用户页是展示你前台真实操作后产生的推荐和画像变化
- 管理员模拟日志依然存在，但它服务于平台趋势和商家分析，不再伪装成你的个人画像
